System Architecture

The LTSU Fee Portal is a single-page application (SPA) built with React 19.2.0, bundled by Vite 7.3.1, styled with Tailwind CSS v4 and a custom CSS design system, and powered by a Supabase backend. The application follows a client-heavy architecture where all rendering, routing, and state management occurs in the browser, with Supabase providing the backend infrastructure without any custom server code (aside from a single Edge Function for admin operations).

The system is composed of four distinct architectural layers:

  • Presentation Layer: React components organized into pages, layout wrappers, UI primitives, and feature-specific modules (admin, dashboard, forms). Tailwind CSS v4 handles utility-first styling, supplemented by a custom CSS design system for the glassmorphism aesthetic.
  • State Management Layer: Three React Context providers (AuthContext, DataContext, ToastContext) manage global application state. Custom hooks (useFeeRequests, useActivityLog, useDebounce) encapsulate complex data-fetching logic with proper cleanup and memoization.
  • Data Access Layer: A single Supabase client instance initialized in lib/supabase.js handles all communication with the backend, including authentication, database queries, real-time subscriptions, and Edge Function invocations.
  • Backend Layer: Supabase provides PostgreSQL as the database engine, Supabase Auth for authentication and session management, Row-Level Security (RLS) policies for data access control, Edge Functions for privileged operations, and pg_cron for scheduled tasks.

Deployment is on Vercel with SPA rewrites configured in vercel.json, ensuring all client-side routes resolve to index.html for proper React Router handling.

Single-Page Application: The entire application is delivered as a single HTML file with JavaScript bundles. All navigation is handled client-side by React Router v7, with Vercel's rewrite rules ensuring deep links and browser refreshes work correctly. There is no server-side rendering (SSR) or static site generation (SSG).

Routing Configuration

React Router v7 Setup

The application uses React Router v7 with a BrowserRouter wrapping all routes in App.jsx. All eight page components are lazy-loaded using React.lazy(), which means the JavaScript for each page is only downloaded when the user first navigates to that route. A Suspense boundary wraps the Routes component and displays a centered Spinner component while the lazy-loaded chunk is being fetched.

ProtectedRoute Component

The ProtectedRoute component acts as a route guard for authenticated pages. It wraps the /dashboard and /log routes and enforces three conditions before rendering the child component:

  1. Authentication Check: Verifies that a Supabase Auth session exists by checking the user object from AuthContext. If no user is present and the context is not in a loading state, the user is redirected to /login.
  2. Role Verification: Compares the user's profile.role against the allowedRoles prop (an array of strings like ['cr', 'teacher']). If the user's role is not in the allowed list, they are redirected to /login.
  3. Active Status Check: Confirms the user's profile.is_active flag is true. Deactivated accounts are redirected to /login even if their session token is still valid.
function ProtectedRoute({ children, allowedRoles }) {
  const { user, profile, loading } = useAuth();

  if (loading) return <Spinner />;

  if (!user || !profile) return <Navigate to="/login" replace />;

  if (!allowedRoles.includes(profile.role))
    return <Navigate to="/login" replace />;

  if (!profile.is_active)
    return <Navigate to="/login" replace />;

  return children;
}

While the authentication check is in progress (the loading state from AuthContext), a Spinner component is rendered to prevent flashing unauthorized content.

Admin Authentication: The /admin route does NOT use the ProtectedRoute component. Instead, the AdminPage component implements its own authentication flow that validates against the VITE_ADMIN_EMAIL environment variable using Supabase Auth, then stores a session flag. This is a separate authentication path from the CR/Teacher login.

AuthProvider (AuthContext.jsx)

The AuthProvider manages the entire authentication lifecycle by wrapping Supabase Auth's session management. It tracks the currently authenticated user, their enriched profile data, and provides login/logout functions to the rest of the application.

State Management

  • user - The raw Supabase Auth user object (contains id, email, app_metadata, etc.)
  • profile - The enriched user profile fetched from the users table, including assigned sections with their branch, semester, and specialisation data
  • loading - Boolean flag indicating whether the initial auth check is in progress

Duplicate Fetch Prevention

A fetchingRef (React useRef) is used to prevent duplicate profile fetch calls. When Supabase Auth's onAuthStateChange fires, it can trigger multiple times (e.g., INITIAL_SESSION followed by SIGNED_IN). The ref acts as a mutex:

const fetchingRef = useRef(false);

async function fetchProfile(userId) {
  if (fetchingRef.current) return;
  fetchingRef.current = true;
  try {
    // ... fetch profile from Supabase ...
  } finally {
    fetchingRef.current = false;
  }
}

Profile Fetching

The fetchProfile function queries the users table with a complex join chain to populate the user's section assignments with full reference data:

supabase
  .from('users')
  .select(`
    *,
    user_sections (
      section_id,
      sections (
        id, name, is_active,
        semesters ( id, name ),
        branches ( id, name ),
        specialisations ( id, name )
      )
    )
  `)
  .eq('id', userId)
  .single()

This returns the user record enriched with an array of user_sections, each containing the full section object with its associated semester, branch, and specialisation data. The result is stored as the profile state.

Provided Values

Value Type Description
user Object | null Raw Supabase Auth user object
profile Object | null Enriched profile with role, name, and sections array (each with branch/semester/spec data)
loading Boolean True during initial auth check, false once session state is determined
login(email, password) Function Authenticates via supabase.auth.signInWithPassword, fetches profile, verifies role and active status
logout() Function Signs out via supabase.auth.signOut(), clears user and profile state

Login Flow

The login() function performs a multi-step validation:

  1. Calls supabase.auth.signInWithPassword({ email, password })
  2. On success, calls fetchProfile(user.id) to load the user's profile and section assignments
  3. Verifies the user's role matches the expected role (the login page may have a role selector)
  4. Checks the is_active flag to ensure the account has not been deactivated by an admin
  5. If any check fails, calls supabase.auth.signOut() and throws an appropriate error

DataProvider (DataContext.jsx)

The DataProvider fetches and caches all reference data that the application needs to populate dropdowns, validate inputs, and enrich displayed records. This data is fetched once on mount and can be individually refreshed when the admin modifies any table.

Data Fetched on Mount

On component mount, the provider fetches five reference tables from Supabase, each filtered to only include active records (is_active = true):

  • branches - Academic branches/departments (e.g., Computer Science, Electronics)
  • specialisations - Specialisations within branches (e.g., AI/ML, Data Science)
  • sections - Class sections with foreign keys to branch, specialisation, and semester
  • batch_configs - Fee amount configurations per section, including academic year and fee amounts
  • semesters - Academic semesters (e.g., Semester 1, Semester 2)

Refetch Functions

Individual refetch functions are provided for each table so that admin operations can refresh only the data they modified:

const refetchBranches = () => fetchTable('branches', setBranches);
const refetchSpecs = () => fetchTable('specialisations', setSpecs);
const refetchSections = () => fetchTable('sections', setSections);
const refetchBatches = () => fetchTable('batch_configs', setBatches);
const refetchSemesters = () => fetchTable('semesters', setSemesters);

Memoized Filtered Arrays

The provider exposes memoized (via useMemo) arrays that filter the raw data to only active records. This prevents unnecessary re-renders when the raw data arrays change but the active subset remains identical:

  • activeBranches - Branches where is_active === true
  • activeSpecs - Specialisations where is_active === true
  • activeSections - Sections where is_active === true
  • activeSemesters - Semesters where is_active === true
  • activeBatches - Batch configs where is_active === true

Helper Functions

ToastProvider (ToastContext.jsx)

The ToastProvider implements a queue-based notification system with automatic dismissal and a maximum visible count. It renders a fixed-position container in the top-right corner of the viewport.

Implementation Details

  • Maximum visible toasts: 5. When a new toast is added and the count exceeds 5, the queue is trimmed to the last 4 items before appending the new one, ensuring the most recent notification is always visible.
  • Unique IDs: A useRef counter generates monotonically increasing integer IDs for each toast, ensuring stable React keys.
  • Auto-dismiss: Each toast has a configurable duration (default varies by type). A timersRef Map stores setTimeout handles keyed by toast ID, allowing cleanup when toasts are manually dismissed or when the component unmounts.
  • Cleanup: On unmount, all active timers in the timersRef Map are cleared to prevent memory leaks and state updates on unmounted components.

Toast Types

Type Appearance Use Case
success Green accent with check icon Successful operations (fee approved, data saved, export complete)
error Red accent with X icon Failed operations (network error, validation failure, unauthorized action)
warning Amber/yellow accent with alert icon Non-blocking warnings (duplicate submission detected, session expiring)
info Blue accent with info icon Informational notices (data refreshed, batch operation in progress)

Data Flow Patterns

Core Data Flow

Data flows through the application in a predictable, unidirectional pattern. Every data operation follows the same fundamental pipeline: a user interaction triggers a function call in a React component, which invokes a Supabase client method, which communicates with the PostgreSQL backend through Supabase's API gateway, and the response updates React state, causing a re-render.

Step-by-Step Data Pipeline

  1. Supabase Client Initialization (lib/supabase.js): The Supabase client is initialized exactly once using createClient() with the project URL and anonymous key read from import.meta.env. The module validates that both environment variables are present before creating the client. This single instance is imported wherever database access is needed.
  2. Authentication State (AuthContext): On app load, onAuthStateChange listens for session events. When a session exists, it triggers fetchProfile() to load the user's enriched profile. The user and profile objects are stored in React state and made available via context.
  3. Reference Data (DataContext): On mount, DataContext fetches all five reference tables in parallel. This data populates dropdowns in the student form and admin panel, and is used by the dashboard to enrich fee request records with human-readable names.
  4. Page-Level Data Consumption: Pages consume contexts via useContext(AuthContext) and useContext(DataContext) hooks. They may also use custom hooks like useFeeRequests for data that is specific to a particular view.
  5. Custom Hook: useFeeRequests: This hook accepts an array of sectionIds and performs a multi-step data enrichment pipeline:
    • Fetches all students belonging to the given sections from the students table
    • Fetches batch_configs and enriches each student with their applicable fee_amount
    • Fetches fee_requests with joined data (student name, section details)
    • Combines students and fee requests into a unified dataset
    • Uses AbortController for cleanup when the component unmounts or section IDs change
  6. Custom Hook: useActivityLog: Provides paginated, filtered activity log data. Accepts search terms and pagination parameters. Uses sanitized search queries (trimming, escaping special characters) to prevent injection in Supabase text search. Returns paginated results with total count for the pagination UI.
  7. Custom Hook: useDebounce: A utility hook that delays updating a value until a specified period of inactivity. Used primarily with search inputs to prevent excessive Supabase queries on every keystroke. Typically configured with a 300ms delay.
AbortController Pattern: The useFeeRequests hook uses the AbortController API to cancel in-flight Supabase requests when the component unmounts or when the dependency array (section IDs) changes. This prevents state updates on unmounted components and avoids race conditions when rapidly switching between sections.

Fee Request Data Flow (Detailed)

The fee request lifecycle involves data flowing through multiple layers. Here is the complete path for a student submitting a fee request:

StudentFormPage
→
Roll Number Lookup
→
Student Verified
→
Fee Details Entered
→
Supabase Insert
  1. The student enters their roll number on the StudentFormPage
  2. The component queries students table by roll number to verify the student exists and retrieve their section/branch data
  3. The fee amount is looked up from batch_configs based on the student's section
  4. Duplicate detection queries fee_requests to check for existing pending/approved requests for the same student
  5. On submission, a new row is inserted into fee_requests with status pending
  6. A unique reference number is generated and returned to the student for tracking

Dashboard Data Flow (Detailed)

When a CR or Teacher loads the dashboard, the following data fetching sequence occurs:

  1. AuthContext provides the user's profile with their assigned section IDs
  2. useFeeRequests(sectionIds) is called with the section IDs from the profile
  3. The hook fetches students, batch configs, and fee requests in sequence
  4. Data is enriched: each fee request gets the student's name, roll number, branch, and the applicable fee amount
  5. The enriched dataset is split into three tab groups: pending, approved, and rejected
  6. Statistics (total students, pending count, approved count, total amount) are computed from the enriched data
  7. For teachers with multiple sections, the SectionSwitcher component allows changing the active section, which triggers a re-fetch with new section IDs

The Vite configuration defines the build pipeline, development server settings, and production optimizations. The configuration file uses two plugins and several esbuild/Rollup options for production builds.

  • @vitejs/plugin-react - Enables React Fast Refresh during development and JSX transformation for production builds.
  • @tailwindcss/vite - Integrates Tailwind CSS v4 directly into the Vite pipeline, enabling JIT compilation and hot-reloading of utility classes without a separate PostCSS configuration.

Production Build Optimizations

In production mode, the build configuration strips debugging artifacts and splits the output into optimized chunks:

defineConfig({
  esbuild: {
    drop: ['debugger'],
    pure: [
      'console.log',
      'console.debug',
      'console.info',
      'console.warn'
    ]
  },
  build: {
    sourcemap: false,
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['react', 'react-dom', 'react-router-dom'],
          supabase: ['@supabase/supabase-js']
        }
      }
    }
  }
})

Build Optimization Details

Option Value Purpose
esbuild.drop ['debugger'] Removes all debugger statements from production output, preventing accidental breakpoints in deployed code
esbuild.pure ['console.log', 'console.debug', 'console.info', 'console.warn'] Tree-shakes all specified console methods from production output. These calls are treated as side-effect-free and eliminated during dead code removal. Note: console.error is intentionally preserved.
build.sourcemap false Disables source map generation in production, reducing build output size and preventing source code exposure in browser DevTools
manualChunks.vendor ['react', 'react-dom', 'react-router-dom'] Bundles React core libraries into a dedicated vendor chunk. These rarely change between deployments, so they benefit from long-term browser caching.
manualChunks.supabase ['@supabase/supabase-js'] Isolates the Supabase client SDK into its own chunk. Separating it from vendor code means Supabase SDK updates do not invalidate the React vendor cache.
Console.error Preserved: The console.error method is intentionally NOT included in the esbuild.pure array. This ensures that genuine runtime errors are still logged in production, allowing developers to diagnose issues in deployed environments via browser DevTools or error monitoring tools.

Deployment Architecture

Vercel SPA Configuration

The application is deployed on Vercel as a static SPA. The vercel.json configuration file contains a single rewrite rule that directs all incoming requests to /index.html, allowing React Router to handle client-side routing:

{
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}

This rewrite is essential for SPA behavior. Without it, direct navigation to routes like /dashboard or /admin would return a 404 from Vercel's static file server, since no physical dashboard/index.html file exists in the build output. The rewrite ensures the React application is always served, and React Router matches the URL to the correct page component.

Vercel's build pipeline runs pnpm install followed by pnpm run build (which executes vite build). The resulting dist/ directory is deployed to Vercel's global edge network. Environment variables (VITE_SUPABASE_URL, VITE_SUPABASE_ANON_KEY, etc.) are configured in the Vercel project dashboard and are embedded into the JavaScript bundles at build time by Vite's import.meta.env replacement.

Edge Network Distribution: Vercel automatically distributes the static build output across its global edge network. Users receive the application from the nearest edge node, minimizing latency. Since the application is entirely client-side (no SSR), there are no serverless function cold starts to contend with for page loads.
