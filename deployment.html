Supabase Backend

The LTSU Fee Portal is split into two independently hosted layers. The frontend is a static single-page application built with Vite and deployed to Vercel. The backend is entirely managed by Supabase, which provides hosted PostgreSQL, authentication, Edge Functions, and Realtime subscriptions.

Frontend: Vercel

Vercel serves the production-built static files from the dist/ directory. It handles SPA routing through a rewrite rule that directs all paths to index.html, allowing React Router to manage client-side navigation. Every push to the main branch on GitHub triggers an automatic rebuild and deployment.

Backend: Supabase

Supabase hosts the entire backend infrastructure. This includes a PostgreSQL database with Row-Level Security policies, Supabase Auth for email/password authentication, Edge Functions (Deno runtime) for privileged admin operations that require the service_role key, and Realtime subscriptions for live data updates on the dashboard.

Vercel Configuration

vercel.json

The vercel.json file at the project root configures Vercel's routing behavior:

{
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}

This rewrite rule ensures ALL routes serve index.html, enabling React Router's client-side routing. Without this configuration, direct navigation to paths like /dashboard or /admin would return a 404 error because those paths do not correspond to physical files on the server. The rewrite intercepts every request and serves the SPA entry point, allowing React Router to parse the URL and render the correct component.

Critical: Without the rewrite rule, refreshing the browser on any route other than / will produce a 404 error. This is the most common deployment issue with single-page applications on static hosting platforms.

Vercel Deployment Steps

  1. Connect GitHub Repository: Link the GitHub repository (Ns81000/LTSU_Fee_Portal) to a new Vercel project. Vercel will automatically detect the repository structure.
  2. Framework Preset: Select Vite as the framework. Vercel auto-detects this from vite.config.js in most cases.
  3. Build Command: pnpm run build (resolves to vite build).
  4. Output Directory: dist — this is where Vite places the production bundle.
  5. Install Command: pnpm install — Vercel runs this before the build step to install all dependencies.
  6. Environment Variables: Set all required VITE_ prefixed environment variables in the Vercel dashboard under Project Settings → Environment Variables. These must be configured before the first deployment.

Build Process

Vite builds the React application with the following configuration defined in vite.config.js:

Plugins

  • @vitejs/plugin-react — Enables React Fast Refresh for hot module replacement during development.
  • @tailwindcss/vite — Tailwind CSS v4 plugin for utility-first CSS processing.

Production Optimizations

When building in production mode (mode === 'production'), the following esbuild optimizations are applied:

Optimization Configuration Effect
Console stripping pure: ['console.log', 'console.debug', 'console.info', 'console.warn'] Removes all non-error console statements from production bundle
Debugger stripping drop: ['debugger'] Removes all debugger statements
Sourcemaps sourcemap: false Disabled in production to reduce bundle size and prevent source exposure
console.error Preserved: Note that console.error is intentionally excluded from the pure list. Error logging remains active in production to support debugging of critical failures. The logAction utility falls back to console.error when an activity log insert fails.

Manual Chunks

The build configuration splits the bundle into manual chunks for optimal caching:

Chunk Name Included Packages Rationale
vendor react, react-dom, react-router-dom Core framework libraries that rarely change; cached long-term by the CDN
supabase @supabase/supabase-js Supabase client SDK; isolated so app code changes do not invalidate this chunk

Output

The build produces a dist/ folder containing index.html and an assets/ directory with hashed JavaScript and CSS files. Vercel serves these files with immutable cache headers on hashed assets and short-lived cache on index.html.

Database Setup Steps

The database must be configured in a specific order due to foreign key dependencies between tables. Follow these steps in sequence:

  1. Create all 10 tables with correct schemas:

    The following tables must be created in the Supabase SQL Editor, respecting foreign key order:

    Table Purpose Dependencies
    users Staff and admin accounts linked to auth.users None (references auth.users via id)
    branches Academic branches (e.g., CSE, ECE) None
    specialisations Branch-specific specialisations branches
    semesters Available semesters per branch branches
    sections Class sections under branches branches
    batch_configs Session year and fee amount configuration None
    students Student master records branches, specialisations, sections
    fee_requests Fee payment submissions and approvals students, semesters, sections, users
    activity_logs Audit trail of all staff/admin actions users
    user_sections Many-to-many mapping of staff to sections users, sections
  2. Enable Row-Level Security (RLS) on all tables:

    Every table must have RLS enabled. Without RLS, the Supabase anon key would grant unrestricted access to all data. Run ALTER TABLE table_name ENABLE ROW LEVEL SECURITY; for each table.

  3. Create RLS policies for each table:

    Define SELECT, INSERT, UPDATE, and DELETE policies that restrict access based on the authenticated user's role (admin, teacher, cr) resolved from the users table. Public tables like branches, semesters, sections, and specialisations allow anonymous SELECT for the student submission form.

  4. Create SECURITY DEFINER RPC functions:

    These PostgreSQL functions execute with elevated privileges regardless of the caller's RLS context:

    • check_duplicate_fee_request — Checks if a student already has a pending/approved request for the same semester
    • submit_fee_request — Inserts a new fee request with reference number generation
    • track_fee_request — Allows anonymous tracking by reference number and roll number
    • batch_approve_requests — Approves multiple fee requests in a single transaction with optimistic locking
  5. Create the reference number generation trigger:

    A PostgreSQL trigger on the fee_requests table automatically generates a unique reference number in the format LTSU-XXXXXX (6-character alphanumeric) on INSERT.

  6. Set up indexes on foreign key columns:

    Create indexes on all foreign key columns to optimize JOIN performance. This includes fee_requests.student_id, fee_requests.semester_id, fee_requests.section_id, students.branch_id, students.spec_id, students.section_id, and others.

Supabase Auth Configuration

  • Enable Email/Password auth provider: In the Supabase Dashboard, navigate to Authentication → Providers and enable the Email provider. This is the only authentication method used by the portal.
  • Disable email confirmation: For simplicity, disable the "Confirm email" requirement under Authentication → Settings. Alternatively, configure a custom SMTP server to send confirmation emails. Admin-created staff accounts are created via the Edge Function which bypasses email confirmation entirely.
  • Set password requirements: Configure minimum password length and complexity under Authentication → Settings to match your institution's security policy.

Admin Account Seeding

The initial admin account must be created manually before the portal can be used:

  1. Create auth user via Supabase Dashboard or CLI:

    Navigate to Authentication → Users → Add User, or use the Supabase CLI. Set the email to admin@ltsu.edu.in and assign a secure password.

  2. Insert into public.users table: 
    INSERT INTO public.users (id, role, full_name, staff_id, email, is_active)
VALUES (
  '<auth-user-uuid>',
  'admin',
  'Admin',
  'ADMIN001',
  'admin@ltsu.edu.in',
  true
);

    Replace <auth-user-uuid> with the UUID generated by Supabase Auth in the previous step. This links the auth.users entry to the public.users profile with the admin role.

Important: The admin account must exist in both auth.users and public.users tables. If the public.users row is missing, the login will succeed at the auth level but the application will fail to resolve the user's role, resulting in a blank screen or redirect loop.

Edge Function Deployment

Edge Function Overview

The portal uses a single Supabase Edge Function located at supabase/functions/admin-actions/index.ts. This function handles privileged operations that require the service_role key, which bypasses Row-Level Security.

Property Value
Runtime Deno (Supabase Edge Functions)
File Path supabase/functions/admin-actions/index.ts
Authentication Uses service_role key (bypasses RLS)
Supported Actions create_user, reset_password, deactivate_user
CORS Configured for cross-origin requests from the Vercel-hosted frontend

Deployment Command

supabase functions deploy admin-actions

This command packages and deploys the Edge Function to the Supabase project. Ensure the Supabase CLI is installed and authenticated with your project before running this command.

Edge Function Actions Detail

1. create_user

Creates a new staff account by performing three operations in sequence:

  1. Creates an entry in auth.users via supabase.auth.admin.createUser() with email and password
  2. Inserts a corresponding record in public.users with role, full_name, staff_id, and email
  3. Inserts user_sections entries to assign the staff member to their designated sections

This action is invoked by the StaffManager component when an admin creates a new staff account.

2. reset_password

Updates the password for an existing auth user via supabase.auth.admin.updateUserById(). This is used by the StaffManager when an admin needs to reset a staff member's password without requiring the old password.

3. deactivate_user

Marks a user as inactive in both auth.users (bans the user) and public.users (sets is_active = false). This effectively locks the account and prevents future logins. Used when an admin deletes or deactivates a staff account.

Edge Function Secrets

No Additional Configuration Needed: The SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY environment variables are automatically available inside Supabase Edge Functions. They are injected by the Supabase runtime and do not need to be manually configured as secrets.

pg_cron Automated Maintenance

Activity Log Cleanup

The activity_logs table records every staff and admin action in the system. Without regular cleanup, this table grows indefinitely and can impact query performance over time. A pg_cron job handles automatic pruning.

Property Value
Job Name cleanup-old-activity-logs
Schedule Daily at 3:00 AM UTC
Cron Expression 0 3 * * *
SQL Statement DELETE FROM public.activity_logs WHERE created_at < now() - interval '31 days'
Retention Period 31 days of audit history

Setting Up pg_cron

  1. Enable the pg_cron extension:

    In the Supabase Dashboard, navigate to Database → Extensions. Search for pg_cron and enable it. This extension allows scheduling PostgreSQL commands to run on a cron schedule directly inside the database.

  2. Create the cron job via SQL Editor: 
    SELECT cron.schedule(
  'cleanup-old-activity-logs',
  '0 3 * * *',
  $$DELETE FROM public.activity_logs WHERE created_at < now() - interval '31 days'$$
);

    This schedules the cleanup job to run every day at 3:00 AM UTC. The $$ dollar-quoting syntax is used to avoid escaping single quotes inside the SQL statement.

Monitoring: You can verify the cron job is registered by querying SELECT * FROM cron.job; in the SQL Editor. To check execution history, query SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 10;.
Retention Period: The 31-day retention window means that any activity log entries older than 31 days are permanently deleted. If your institution requires longer audit trails for compliance, adjust the interval accordingly before creating the cron job.

EmailJS Setup

EmailJS Configuration Steps

EmailJS is used to send approval and rejection email notifications to students directly from the browser. No server-side email infrastructure is required.

  1. Create an EmailJS account:

    Sign up at emailjs.com. The free tier allows 200 emails per month, which is sufficient for most use cases.

  2. Create an Email Service:

    Connect your email provider (Gmail, Outlook, etc.) as a service. The service ID for this project is service_va1s1tq.

  3. Create the Approval Template:

    Template ID: template_bsick8e. Configure the template with the following dynamic variables:

    Variable Description
    to_email Student's email address
    student_name Full name of the student
    reference_number Fee request reference number (LTSU-XXXXXX)
    fee_amount Approved fee amount
    semester Semester name
    approved_date Date of approval
    approved_by Name of the staff who approved
  4. Create the Rejection Template:

    Template ID: template_wd7lk49. Configure the template with the following dynamic variables:

    Variable Description
    to_email Student's email address
    student_name Full name of the student
    reference_number Fee request reference number (LTSU-XXXXXX)
    rejection_reason Reason for rejection provided by staff
    rejected_date Date of rejection
  5. Get the Public Key:

    Navigate to Account → API Keys in the EmailJS dashboard. The public key for this project is Hp31LVJ0_HZsNpnN4. This key is used in the client-side emailjs.send() call.

  6. Add credentials to environment variables:

    Set VITE_EMAILJS_SERVICE_ID, VITE_EMAILJS_APPROVAL_TEMPLATE_ID, VITE_EMAILJS_REJECTION_TEMPLATE_ID, and VITE_EMAILJS_PUBLIC_KEY in both the local .env file and the Vercel dashboard.

Local Development

Development Setup

# Clone the repository
git clone https://github.com/Ns81000/LTSU_Fee_Portal.git
cd LTSU_Fee_Portal

# Install dependencies
pnpm install

# Configure environment variables
cp .env.example .env
# Edit .env with your Supabase and EmailJS credentials

# Start the development server
pnpm run dev
# Opens at http://localhost:5173

# Build for production
pnpm run build

# Preview the production build locally
pnpm run preview
Hot Module Replacement: The development server uses Vite's HMR with React Fast Refresh. Changes to React components are reflected instantly without a full page reload. CSS changes via Tailwind are also hot-reloaded.
Environment File Required: The .env file must be present in the project root before starting the dev server. Without it, all Supabase and EmailJS calls will fail with undefined credentials. Copy from .env.example and fill in the actual values.
