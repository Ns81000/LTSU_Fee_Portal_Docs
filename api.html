API Overview

The LTSU Fee Portal communicates with Supabase in three distinct ways. Each method serves a different purpose and carries different security characteristics regarding authentication and Row Level Security (RLS) bypass behaviour.

  1. Supabase Client SDK — Direct table queries (select, insert, update, delete) issued from the browser via the @supabase/supabase-js client library. These operations run under the authenticated user's JWT and are subject to all RLS policies defined on each table.
  2. RPC Functions — PostgreSQL stored procedures invoked via supabase.rpc(). These are written in PL/pgSQL and defined with SECURITY DEFINER, meaning they execute with the privileges of the function creator and bypass RLS.
  3. Edge Functions — Deno-based serverless functions deployed to the Supabase Edge runtime. Called via supabase.functions.invoke(), they create an internal admin client using the service role key and therefore bypass all RLS policies.

Summary Table

Type Count Auth Required Bypasses RLS
Client Queries ~30 Yes (JWT) No
RPC Functions 4 No (SECURITY DEFINER) Yes
Edge Functions 1 (3 actions) Yes (Admin JWT) Yes
Note: Client Queries are the most common communication method, accounting for approximately 30 distinct query patterns across the codebase. RPC Functions handle the four most security-sensitive operations where bypassing RLS is intentional and necessary. The single Edge Function groups three admin-only actions behind one HTTP endpoint.

Supabase Client Setup

File: src/lib/supabase.js

The Supabase client is created once at module scope and exported as a singleton. Every file in the application imports this same instance, ensuring a single connection is reused throughout the session.

import { createClient } from '@supabase/supabase-js'

const supabaseUrl = import.meta.env.VITE_SUPABASE_URL
const supabaseAnonKey = import.meta.env.VITE_SUPABASE_ANON_KEY

export const supabase = createClient(supabaseUrl, supabaseAnonKey)

The client instance automatically attaches the user's JWT (stored in localStorage by the Supabase auth module) to all outgoing requests. This means every query, insert, update, and delete operation is authenticated without any manual header management.

Environment Variables: Both VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY are prefixed with VITE_ because Vite only exposes environment variables with that prefix to client-side code. These values are safe to expose publicly — the anon key is a public key that relies on RLS for security.

CORS Configuration

The Edge Function handles CORS preflight requests to allow cross-origin calls from the Vercel-hosted frontend. When an OPTIONS request arrives, the function returns the following headers immediately:

  • Access-Control-Allow-Origin: *
  • Access-Control-Allow-Headers: authorization, x-client-info, apikey, content-type
  • Access-Control-Allow-Methods: POST

These headers are also applied to all non-preflight responses to ensure browsers accept the response body.

Internal Admin Client

Inside the Edge Function, a separate Supabase client is created using the SUPABASE_URL and SUPABASE_SERVICE_ROLE_KEY environment variables. These are auto-injected by the Supabase runtime and never exposed to the browser. The service role client bypasses all RLS policies, which is necessary for operations like creating auth users and managing profiles across the system.

const supabaseAdmin = createClient(
  Deno.env.get('SUPABASE_URL'),
  Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')
)

Client-side Invocation Pattern

All three actions are invoked using the same supabase.functions.invoke() method. The action field in the request body determines which operation the Edge Function performs.

const { data, error } = await supabase.functions.invoke('admin-actions', {
  body: { action: 'create_user', email, password, full_name, role, staff_id, sections }
})

Action: create_user

Purpose: Create a new staff account (teacher or CR) with an auth user entry, a public profile row, and section assignments.

Request Body

{
  "action": "create_user",
  "email": "teacher@example.com",
  "password": "GenerAted12!",
  "full_name": "John Smith",
  "role": "teacher",
  "staff_id": "TCH001",
  "sections": ["uuid-1", "uuid-2"]
}

Request Body Fields

Field Type Required Description
action string Yes Must be "create_user"
email string Yes Valid email address for the new account
password string Yes Password (generated client-side, minimum 8 characters)
full_name string Yes Display name for the staff member
role string Yes Either "teacher" or "cr"
staff_id string Yes Unique staff identifier (e.g., TCH001)
sections string[] Yes Array of section UUIDs to assign to this user

Process

  1. Creates an auth.users entry via supabaseAdmin.auth.admin.createUser({ email, password, email_confirm: true }). The email_confirm: true flag skips the email verification step, since admin-created accounts are trusted.
  2. Inserts a row into public.users with the following fields: { id: authUser.id, email, full_name, role, staff_id, is_active: true }. The id is linked to the auth user's UUID.
  3. For each section UUID in the sections array, inserts a row into user_sections: { user_id: authUser.id, section_id: sectionUUID }.
  4. Returns: { user: publicUser }
Receive create_user request
→
Create auth.users entry
→
Insert public.users row
→
Insert user_sections rows
→
Return { user }

Error Handling

  • Auth user creation fails: Returns the error message immediately. No cleanup is needed since nothing was created yet.
  • public.users insert fails: Attempts to clean up the auth user that was just created to avoid orphaned auth entries.
  • user_sections insert fails: Logs a warning to the console but does not roll back the auth user or profile. The user is created but may have incomplete section assignments.
No Transactional Guarantee: Because the Edge Function performs three separate operations (auth user, profile, sections), there is no database-level transaction wrapping all three. If the profile insert succeeds but sections fail, the user exists with no section assignments. Admin must manually fix this via the Staff Manager page.

Action: reset_password

Purpose: Reset a staff member's password. This is an admin-initiated operation — there is no self-service password reset flow in the portal.

Request Body

{
  "action": "reset_password",
  "user_id": "uuid",
  "password": "NewPassw0rd!"
}

Request Body Fields

Field Type Required Description
action string Yes Must be "reset_password"
user_id string Yes UUID of the target user in auth.users
password string Yes New password (minimum 8 characters)

Process

  1. Calls supabaseAdmin.auth.admin.updateUserById(user_id, { password }) to update the password in the auth.users table.
  2. Returns: { success: true }
Session Impact: Resetting a password does not immediately invalidate existing sessions. The user's current JWT remains valid until it expires. However, the next login attempt will require the new password.

Action: deactivate_user

Purpose: Deactivate (soft-delete) a staff account. The user's data is preserved but they can no longer log in or access the system.

Request Body

{
  "action": "deactivate_user",
  "user_id": "uuid"
}

Request Body Fields

Field Type Required Description
action string Yes Must be "deactivate_user"
user_id string Yes UUID of the target user

Process

  1. Updates public.users SET is_active = false WHERE id = user_id. This soft-deletes the profile.
  2. Updates the auth user via supabaseAdmin.auth.admin.updateUserById(user_id, { ban_duration: 'none' }) to ban the user at the auth level, preventing any future login attempts.
  3. Returns: { success: true }
Soft Delete Only: Deactivation does not remove any data from the database. The user's profile, section assignments, and any fee requests they reviewed remain intact for audit purposes. The is_active flag is checked by the AuthContext on login to prevent deactivated users from accessing the dashboard.

RPC Functions (PostgreSQL Stored Procedures)

All four RPC functions are defined with SECURITY DEFINER, meaning they execute with the privileges of the function creator (typically the postgres role) rather than the calling user. This allows them to bypass Row Level Security policies, which is necessary for operations that span multiple tables or require unrestricted access to the data.

Function Name Called From Returns Purpose
check_duplicate_fee_request StudentForm (Step 2) TEXT Check for existing fee request
submit_fee_request StudentForm (Step 2) TEXT Insert new fee request
track_fee_request TrackingPage TABLE Look up request by reference number
batch_approve_requests Dashboard INTEGER Bulk approve pending requests

check_duplicate_fee_request

Purpose: Check if a student already has a fee request for a specific semester. Called from the Student Form (Step 2) before submission to prevent duplicate requests.

SQL Signature

CREATE OR REPLACE FUNCTION check_duplicate_fee_request(
  p_roll_number TEXT,
  p_semester INTEGER
) RETURNS TEXT
LANGUAGE plpgsql
SECURITY DEFINER

Parameters

Parameter Type Description
p_roll_number TEXT The student's roll number
p_semester INTEGER The semester number to check against

Logic

  1. Queries fee_requests WHERE roll_number = p_roll_number AND semester = p_semester.
  2. If a row exists with status = 'pending', returns 'pending'.
  3. If a row exists with status = 'approved', returns 'approved'.
  4. If a row exists with status = 'rejected', returns 'rejected'.
  5. If no matching row exists, returns 'none'.

Client Invocation

const { data, error } = await supabase.rpc('check_duplicate_fee_request', {
  p_roll_number: rollNumber,
  p_semester: semester
})
// data = 'none' | 'pending' | 'approved' | 'rejected'

Behaviour in UI

Return Value UI Behaviour
'none' Proceed with submission normally
'pending' Show warning: "You already have a pending request for this semester"
'approved' Show info: "Your fee for this semester is already approved"
'rejected' Allow resubmission (the student can submit a new request)

submit_fee_request

Purpose: Submit a new fee request with all validation performed server-side. Called from the Student Form (Step 2) on form submit.

SQL Signature

CREATE OR REPLACE FUNCTION submit_fee_request(
  p_student_name TEXT,
  p_roll_number TEXT,
  p_branch TEXT,
  p_specialisation TEXT,
  p_section TEXT,
  p_semester INTEGER,
  p_email TEXT,
  p_fee_amount NUMERIC,
  p_batch TEXT
) RETURNS TEXT
LANGUAGE plpgsql
SECURITY DEFINER

Parameters

Parameter Type Description
p_student_name TEXT Full name of the student
p_roll_number TEXT Student's roll number
p_branch TEXT Branch name (e.g., "Computer Science")
p_specialisation TEXT Specialisation name within the branch
p_section TEXT Section name (e.g., "A")
p_semester INTEGER Semester number

p_fee_amount NUMERIC Fee amount in INR

  1. The reference number is generated automatically via the generate_reference_number BEFORE INSERT trigger (see Reference Number Generation section below).
  2. Inserts a new row into fee_requests with status = 'pending' and receipt_sent = false.
  3. Returns the generated reference_number string (e.g., LTSU-2025-00042).

const { data, error } = await supabase.rpc('submit_fee_request', {
  p_student_name: name,

  p_branch: branch,
  p_specialisation: spec,
  p_section: section,
  p_semester: semester,
  p_email: email,
  p_fee_amount: feeAmount,
  p_batch: batch
})
// data = 'LTSU-2025-00042' (reference number)

track_fee_request

Purpose: Look up a fee request by its reference number for the public tracking page. Students use this to check the status of their submission without logging in.

SQL Signature

CREATE OR REPLACE FUNCTION track_fee_request(
  p_reference_number TEXT
) RETURNS TABLE (
  reference_number TEXT,
  student_name TEXT,
  roll_number TEXT,
  branch TEXT,
  specialisation TEXT,
  section TEXT,
  semester INTEGER,
  email TEXT,
  fee_amount NUMERIC,
  status TEXT,
  rejection_reason TEXT,
  created_at TIMESTAMPTZ,
  updated_at TIMESTAMPTZ,
  reviewed_by TEXT,
  batch TEXT
)
LANGUAGE plpgsql
SECURITY DEFINER

Parameters

Parameter Type Description
p_reference_number TEXT The reference number to search for (e.g., LTSU-2025-00042)

Logic

  1. Queries fee_requests WHERE reference_number = p_reference_number.
  2. If found, returns the full row as a table result with all 15 columns.
  3. If not found, returns an empty result set (zero rows).

Client Invocation

const { data, error } = await supabase.rpc('track_fee_request', {
  p_reference_number: refNumber
})
// data = [] (not found) or [{ reference_number, student_name, ... }]
Public Access: This RPC function is one of the few operations that does not require authentication. Students can track their request status using only their reference number, which serves as a form of "knowledge-based" authentication.

batch_approve_requests

Purpose: Approve multiple fee requests at once in a single database operation. Called from the Dashboard "Batch Approve" feature.

SQL Signature

CREATE OR REPLACE FUNCTION batch_approve_requests(
  p_request_ids UUID[],
  p_reviewed_by TEXT
) RETURNS INTEGER
LANGUAGE plpgsql
SECURITY DEFINER

Parameters

Parameter Type Description
p_request_ids UUID[] Array of fee request UUIDs to approve
p_reviewed_by TEXT Name of the staff member performing the approval

Logic

  1. Executes: UPDATE fee_requests SET status='approved', reviewed_by=p_reviewed_by, updated_at=now() WHERE id = ANY(p_request_ids) AND status='pending'
  2. Returns the count of actually updated rows as an integer.
  3. Only updates rows that are still in 'pending' status. This implements optimistic concurrency control — if another staff member already approved or rejected a request while the batch was being prepared, those rows are silently skipped.

Client Invocation

const { data, error } = await supabase.rpc('batch_approve_requests', {
  p_request_ids: selectedIds,  // array of UUIDs
  p_reviewed_by: user.full_name
})
// data = 5 (number of approved requests)
Maximum Batch Size: The client-side interface enforces a maximum batch size of 25 requests per operation. This limit is checked in the UI before calling the RPC function and is not enforced at the database level.

Client-Side Query Patterns

The following sections document the major Supabase Client SDK query patterns used throughout the application. All queries run under the authenticated user's JWT and are subject to Row Level Security policies.

DataProvider Queries

File: src/context/DataContext.jsx

The DataProvider context loads all reference data on mount. This data is used across the application for dropdown options, form validation, and display purposes. Each query fetches data from a reference table and is executed once when the provider mounts.

Branches

const { data } = await supabase
  .from('branches')
  .select('*')
  .order('name')

Specialisations (with branch join)

const { data } = await supabase
  .from('specialisations')
  .select('*, branches(name)')
  .order('name')

Semesters

const { data } = await supabase
  .from('semesters')
  .select('*')
  .order('number')

Sections (with multiple joins)

const { data } = await supabase
  .from('sections')
  .select('*, branches(name), specialisations(name), semesters(number)')
  .order('name')

Batch Configs

const { data } = await supabase
  .from('batch_configs')
  .select('*')
  .order('year')
Join Syntax: Supabase uses PostgREST's embedded resource syntax for joins. Writing branches(name) inside the select string tells PostgREST to follow the foreign key relationship and embed the name column from the related branches table as a nested object.

Fee Requests Queries

File: src/hooks/useFeeRequests.js

The useFeeRequests hook fetches fee requests scoped to specific sections assigned to the current user. Requests are filtered by branch, specialisation, and section name, then sorted by creation date (newest first).

const { data } = await supabase
  .from('fee_requests')
  .select('*')
  .eq('branch', sectionBranch)
  .eq('specialisation', sectionSpec)
  .eq('section', sectionName)
  .order('created_at', { ascending: false })

This query chains three .eq() filters, each translated to a WHERE clause by PostgREST. The result is an array of fee request objects matching the given section's branch, specialisation, and section name.

Students Query

File: src/components/admin/StudentManager.jsx

The Student Manager component fetches paginated, searchable, filterable, and sortable student data. This is the most complex client-side query in the application.

const { data, count } = await supabase
  .from('students')
  .select('*', { count: 'exact' })
  .ilike('name', `%${search}%`)
  .eq('branch', filterBranch)
  .range(from, to)
  .order(sortCol, { ascending: sortDir === 'asc' })

Query Breakdown

Method Purpose SQL Equivalent
.select('*', { count: 'exact' }) Fetch all columns and return total count for pagination SELECT *, COUNT(*) OVER()
.ilike('name', '%...%') Case-insensitive partial match on student name WHERE name ILIKE '%search%'
.eq('branch', filterBranch) Exact match on branch column AND branch = 'value'
.range(from, to) Pagination (zero-indexed row range) LIMIT ... OFFSET ...
.order(sortCol, { ascending }) Dynamic sort column and direction ORDER BY col ASC/DESC

The count value returned alongside data is used to calculate total pages for the pagination controls. The .range() method accepts zero-indexed start and end positions (inclusive).

Activity Logging

File: src/utils/logAction.js

The logAction utility function writes audit trail entries to the activity_logs table. It is called throughout the application whenever a significant action occurs.

export async function logAction(action, details = {}, userId = null) {
  const sanitized = stripPII(details)
  await supabase.from('activity_logs').insert({
    action,
    details: sanitized,
    performed_by: userId
  })
}

Parameters

Parameter Type Description
action string Action type identifier (30+ types, e.g., 'fee_request_approved')
details object Arbitrary metadata object describing the action context
userId string | null UUID of the user who performed the action (null for anonymous actions)

PII Stripping

The internal stripPII function processes the details object before insertion. It removes sensitive fields including email addresses and passwords from the logged data. This ensures that even if a developer passes the full request body to logAction, no personally identifiable information is stored in the activity log.

Fire-and-Forget: The logAction function intentionally does not await the result or handle errors from the insert operation at the call site. Activity logging failures are silent — they do not interrupt the user's workflow or display error messages.

Approve Single Request

The single-request approval pattern uses optimistic concurrency control to prevent race conditions when multiple staff members review the same request simultaneously.

const { data, error } = await supabase
  .from('fee_requests')
  .update({
    status: 'approved',
    reviewed_by: user.full_name,
    updated_at: new Date().toISOString()
  })
  .eq('id', requestId)
  .eq('status', 'pending')    // Optimistic concurrency control
  .select()

Concurrency Control

The second .eq('status', 'pending') clause is the key to the concurrency model. It means the UPDATE will only match rows that are still in 'pending' status at the moment the query executes. If another staff member has already approved or rejected the request, the WHERE clause will not match any rows, and data will be an empty array.

  • If data.length === 1: The approval succeeded. The request was still pending and has now been approved.
  • If data.length === 0: The request was already processed by someone else. The UI displays a message indicating the row was already reviewed.

Reject Single Request

The rejection pattern mirrors the approval pattern with the addition of a rejection_reason field. The same optimistic concurrency control applies.

const { data, error } = await supabase
  .from('fee_requests')
  .update({
    status: 'rejected',
    reviewed_by: user.full_name,
    rejection_reason: reason,
    updated_at: new Date().toISOString()
  })
  .eq('id', requestId)
  .eq('status', 'pending')
  .select()

The rejection_reason string is collected from the user via the RejectModal component, which enforces a minimum length before allowing submission. This ensures every rejection has a documented reason for the student to review on the tracking page.

Reference Number Generation (Database Trigger)

The reference number is generated automatically by a PostgreSQL trigger that fires before every insert into the fee_requests table. This ensures reference numbers are sequential, unique, and never assigned client-side.

Trigger Definition

CREATE OR REPLACE FUNCTION generate_reference_number()
RETURNS TRIGGER AS $$
DECLARE
  year_part TEXT;
  seq_num INTEGER;
  new_ref TEXT;
BEGIN
  year_part := to_char(now(), 'YYYY');

  SELECT COALESCE(
    MAX(CAST(SUBSTRING(reference_number FROM 10) AS INTEGER)), 0
  ) + 1
  INTO seq_num
  FROM fee_requests
  WHERE reference_number LIKE 'LTSU-' || year_part || '-%';

  new_ref := 'LTSU-' || year_part || '-' || LPAD(seq_num::TEXT, 5, '0');
  NEW.reference_number := new_ref;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

Format

Reference numbers follow the pattern LTSU-YYYY-XXXXX where:

  • LTSU — Fixed prefix identifying the system
  • YYYY — Four-digit calendar year (e.g., 2025)
  • XXXXX — Five-digit zero-padded sequential number (e.g., 00001)

Sequence Behaviour

Scenario Result
First request of the year LTSU-2025-00001
42nd request of the year LTSU-2025-00042
New calendar year starts LTSU-2026-00001 (counter resets)
Maximum capacity LTSU-2025-99999 (99,999 requests per year)
Reset Mechanism: The sequence resets automatically each calendar year because the WHERE clause filters by the current year prefix. When January 1st arrives, no existing rows match the new year pattern, so COALESCE returns 0 and the counter starts at 1 again.

Authentication API

File: src/context/AuthContext.jsx

All authentication operations are handled through the Supabase Auth module. The AuthContext provider wraps the entire application and manages the current user's session, profile, and authentication state.

Login

const { data, error } = await supabase.auth.signInWithPassword({
  email,
  password
})

After successful authentication, the context immediately fetches the user's full profile with section assignments:

const { data: profile } = await supabase
  .from('users')
  .select('*, user_sections(section_id, sections(*, branches(name), specialisations(name), semesters(number)))')
  .eq('id', authUser.id)
  .single()

The .single() modifier ensures exactly one row is returned. If the user does not have a corresponding public.users profile (e.g., it was deleted), the query will error and the user will be signed out.

Logout

await supabase.auth.signOut()

Clears the session from localStorage and triggers the onAuthStateChange listener with a SIGNED_OUT event. The AuthContext then resets all state to null and redirects to the login page.

Session Check

const { data: { session } } = await supabase.auth.getSession()

supabase.auth.onAuthStateChange((event, session) => {
  // Handle session changes: SIGNED_IN, SIGNED_OUT, TOKEN_REFRESHED
})

On initial mount, getSession() checks for an existing session in localStorage. The onAuthStateChange listener then handles all subsequent state transitions, including automatic token refresh when the JWT expires.

Error Handling Patterns

All Supabase operations in the LTSU Fee Portal follow a consistent destructured return pattern. The { data, error } tuple is checked immediately after every query, and errors are surfaced to the user via the global toast notification system.

Standard Query Pattern

const { data, error } = await supabase.from('table').select('*')
if (error) {
  toast.error('Failed to load data')
  console.error(error)
  return
}
// Use data

Edge Function Error Pattern

Edge Function calls require checking both the transport-level error (network failures, CORS issues) and the application-level data.error (returned by the function logic):

const { data, error } = await supabase.functions.invoke('admin-actions', { body })
if (error || data?.error) {
  toast.error(data?.error || 'Operation failed')
  return
}
// Use data

Error Response Structure

Source Error Location Example
Client SDK query error object with .message, .code, .details { message: "new row violates RLS", code: "42501" }
RPC function error object with .message { message: "function returned error" }
Edge Function (transport) error from functions.invoke() Network error, CORS failure, 401/500 HTTP status
Edge Function (application) data.error string { error: "User with this email already exists" }
Double-Check Required: When working with Edge Functions, always check both error (from the SDK) and data?.error (from the function). The SDK may return error: null even when the function itself returned a 400 status with an error message in the body. This is a common source of bugs when only one check is implemented.
