EmailJS Integration Overview

The LTSU Fee Portal uses EmailJS (@emailjs/browser) as its email delivery service. EmailJS is a client-side email sending library that communicates directly with the EmailJS API from the browser -- no backend server or serverless function is required. This makes it ideal for a Supabase-backed SPA where all logic runs in the client.

How It Works

EmailJS is initialized once at application startup in main.jsx via a dynamic import. When a CR or Teacher approves or rejects a fee request on the Dashboard, the application constructs a template parameters object and calls emailjs.send() with the appropriate service ID and template ID. EmailJS then renders the HTML email template on its server-side infrastructure and sends it to the student's email address via the configured SMTP provider.

Key Characteristics

  • Library: @emailjs/browser (client-side, no server required)
  • Initialization: emailjs.init(publicKey) called in main.jsx before React renders
  • Service ID: service_va1s1tq (configured via VITE_EMAILJS_SERVICE_ID)
  • Public Key: Hp31LVJ0_HZsNpnN4 (configured via VITE_EMAILJS_PUBLIC_KEY)
  • Two templates: one for approval notifications, one for rejection notifications
  • Trigger point: DashboardPage.jsx -- emails are sent immediately after a CR/Teacher approves or rejects a fee request
  • Non-blocking: email failures do not prevent the approve/reject operation from completing
Why Client-Side? Since the LTSU Fee Portal is a fully client-side React SPA with no custom backend, EmailJS provides a way to send emails without deploying a server or edge function. The public key is exposed in the browser, but EmailJS rate-limits and domain-restricts API usage to prevent abuse.

Environment Variables

All email-related configuration is managed through Vite environment variables (prefixed with VITE_). These are embedded at build time and accessible via import.meta.env in the application code.

VariableValuePurpose
VITE_EMAILJS_SERVICE_ID service_va1s1tq EmailJS service identifier -- links to the SMTP provider configured in the EmailJS dashboard
VITE_EMAILJS_APPROVAL_TEMPLATE_ID template_bsick8e Template ID for the approval/receipt email
VITE_EMAILJS_REJECTION_TEMPLATE_ID template_wd7lk49 Template ID for the rejection notification email
VITE_EMAILJS_PUBLIC_KEY Hp31LVJ0_HZsNpnN4 EmailJS public API key used to authenticate client-side requests
Build-Time Embedding: Because these are VITE_-prefixed variables, they are statically replaced at build time by Vite. Changing a value requires a full rebuild and redeployment. The values are visible in the compiled JavaScript bundle -- this is by design, as EmailJS public keys are intended to be client-facing.

Detailed Approval Flow (Single Request)

When a CR or Teacher clicks the Approve button for a single fee request, the following sequence executes in DashboardPage.jsx:

  1. Status Update: The fee request's status is updated to 'approved' in the fee_requests table via Supabase. The reviewed_at and reviewed_by fields are also set. An optimistic concurrency check ensures the request is still in 'pending' status (.eq('status', 'pending')). If count === 0, the request was already processed by another reviewer, and the operation aborts with a warning toast.
  2. Receipt Generation: A PDF receipt is generated using generateReceipt() from src/utils/generateReceipt.js. This runs in parallel conceptually but completes before the email step.
  3. Template Parameters Construction: An object is built with all the template variables required by the approval email template:
{
  to_email: req.student_email,
  student_name: req.students?.full_name,
  reference_number: req.reference_number,
  roll_number: req.roll_number,
  branch_name: req.students?.branches?.name,
  specialisation_name: req.students?.specialisations?.name,
  section_name: req.students?.sections?.name,
  batch_year: req.batch_year,
  semester_name: req.semesters?.name,
  fee_amount: req.fee_amount,
  approved_by_name: profile.full_name,
  approved_date: formatDate(new Date().toISOString()),
}
  1. Email Send: emailjs.send() is called with the service ID (VITE_EMAILJS_SERVICE_ID), template ID (VITE_EMAILJS_APPROVAL_TEMPLATE_ID), and the parameters object.
  2. receipt_sent Update: On successful email delivery, the receipt_sent flag is set to true on the fee request record: 
    await supabase
  .from('fee_requests')
  .update({ receipt_sent: true })
  .eq('id', req.id)
  3. Activity Log Entry: A receipt_sent action is logged via logAction() with the reference number and email address in details.
  4. Error Handling: If emailjs.send() throws, the error is caught, emailFailed is set to true, and the error is logged to the console. The approve operation itself is not reverted -- the student's fee request remains approved.

Detailed Batch Approval Flow

The batch approve flow uses the batch_approve_requests RPC function to update all selected requests in a single database call. After the RPC succeeds, emails are sent sequentially in a loop:

  1. The batch_approve_requests RPC is called with all selected request IDs, setting their status to approved.
  2. EmailJS is dynamically imported once before the loop.
  3. For each selected request, emailjs.send() is called with the approval template parameters. A 200ms delay (setTimeout) is inserted between each send to avoid EmailJS rate limits.
  4. Successfully emailed request IDs are collected in emailSuccessIds.
  5. After the loop, a single bulk update sets receipt_sent: true for all successfully emailed requests using .in('id', emailSuccessIds).
  6. A toast displays the count of approved requests and successfully sent emails, e.g., "5 request(s) approved. 4 email(s) sent."
const emailjs = await import('@emailjs/browser').then(m => m.default)
let emailSuccessIds = []
const total = selectedRequests.length

for (let i = 0; i < total; i++) {
  const req = selectedRequests[i]
  setBatchProgress(`(${i + 1}/${total})`)
  try {
    await emailjs.send(
      import.meta.env.VITE_EMAILJS_SERVICE_ID,
      import.meta.env.VITE_EMAILJS_APPROVAL_TEMPLATE_ID,
      { /* template params */ }
    )
    emailSuccessIds.push(req.id)
  } catch (emailErr) {
    console.error(`Email failed for ${req.roll_number}:`, emailErr)
  }
  await new Promise(r => setTimeout(r, 200))
}

if (emailSuccessIds.length > 0) {
  await supabase
    .from('fee_requests')
    .update({ receipt_sent: true })
    .in('id', emailSuccessIds)
}

Detailed Rejection Flow

Rejection emails follow a similar pattern to single approvals, but use the rejection template and include a rejection reason:

  1. The fee request's status is updated to 'rejected' with the rejection_reason field populated.
  2. Template parameters are constructed with rejection-specific fields:
{
  to_email: req.student_email,
  student_name: req.students?.full_name,
  reference_number: req.reference_number,
  roll_number: req.roll_number,
  semester_name: req.semesters?.name,
  reviewed_by_name: profile.full_name,
  reviewed_date: formatDate(new Date().toISOString()),
  rejection_reason: reason,
}
  1. emailjs.send() is called with VITE_EMAILJS_REJECTION_TEMPLATE_ID.
  2. On success, a rejection_email_sent action is logged.
  3. On failure, the error is caught and logged to console. The rejection itself is not reverted.
Sequential Email Sending: In batch approve, emails are sent one at a time with a 200ms throttle delay between each. This respects EmailJS rate limits and prevents the service from rejecting burst requests. Progress is displayed to the user via setBatchProgress().

receipt_sent Flag

The receipt_sent column on the fee_requests table is a boolean flag that tracks whether the notification email for a given fee request has been successfully delivered via EmailJS.

Schema

ColumnTypeDefaultDescription
receipt_sent boolean false Indicates whether the approval/rejection email was successfully sent to the student

Behaviour

  • Default value: false -- every new fee request starts with receipt_sent = false
  • Set to true: Only after emailjs.send() resolves successfully. For single approvals, the update is immediate. For batch approvals, all successfully emailed IDs are collected and updated in a single bulk query after the loop.
  • Remains false on email failure: If emailjs.send() throws an error, the flag stays false. This allows potential future retry mechanisms to identify unsent emails.
  • Display in Dashboard: The receipt_sent field is included in the formatters utility (src/utils/formatters.js) where it maps to the label "Receipt email sent" for display purposes in activity logs and exports.
Idempotency Consideration: Currently, the application does not explicitly check receipt_sent before attempting to send an email during the approve/reject flow. However, because the status check (.eq('status', 'pending')) prevents re-processing already-handled requests, duplicate emails are effectively prevented. The flag serves primarily as an audit trail and for potential retry logic.

Error Handling

Email delivery in the LTSU Fee Portal is treated as a non-critical operation. The system is designed so that email failures never block or revert the primary approve/reject workflow.

Error Handling Strategy

ScenarioBehaviour
EmailJS .send() throws Error is caught in a try/catch block. The approve/reject operation is not reverted. The fee request status remains changed.
Console logging The error object is logged via console.error('Email send failed:', emailErr) for single approvals and console.error(`Email failed for ${req.roll_number}:`, emailErr) for batch operations.
Toast notification For single approvals, the toast message adjusts based on emailFailed: if true, the toast includes a warning note about email delivery failure alongside the success confirmation.
receipt_sent flag Remains false when email fails, distinguishing successful deliveries from failures in the database.
Activity log The receipt_sent action is only logged if the email send succeeds. If it fails, no receipt_sent log entry is created, but the request_approved or request_rejected entry is still logged.
Batch approve partial failure Each email in the batch loop is individually wrapped in try/catch. If email fails for one request, the loop continues to the next. The final toast shows both counts: "5 request(s) approved. 3 email(s) sent."

Failure Scenarios

Common reasons for EmailJS send failures include:

  • Rate limiting: EmailJS free tier has monthly send limits. Exceeding the limit causes all subsequent sends to fail.
  • Invalid public key: If VITE_EMAILJS_PUBLIC_KEY is misconfigured or empty, emailjs.init() will initialise with an invalid key and all sends will fail.
  • Network issues: Transient network failures between the client browser and the EmailJS API.
  • Invalid email address: If to_email (the student's email) is malformed, EmailJS may reject the request.
  • Template misconfiguration: If the template ID does not match an existing template in the EmailJS dashboard, the API returns an error.
No Automatic Retry: The current implementation does not include automatic retry logic for failed email sends. If an email fails, the only recovery path is manual intervention (re-approving or using a future admin tool). The receipt_sent = false flag can be used to query for requests that were approved but never emailed.

EmailJS Initialization

EmailJS is initialized at application startup in src/main.jsx, before React's createRoot().render() call. The initialization is performed via a dynamic import to keep the main bundle size minimal.

Initialization Code

import('@emailjs/browser').then(m => {
  m.default.init(import.meta.env.VITE_EMAILJS_PUBLIC_KEY || '')
})

Breakdown

  • Dynamic import: import('@emailjs/browser') returns a Promise. The @emailjs/browser module is loaded asynchronously and does not block the initial bundle parse or React rendering.
  • Module default export: The .then(m => m.default.init(...)) chain accesses the default export of the EmailJS module and calls init() with the public key.
  • Fallback empty string: import.meta.env.VITE_EMAILJS_PUBLIC_KEY || '' ensures that if the environment variable is undefined, an empty string is passed rather than undefined. This prevents a runtime error during init but means all subsequent emailjs.send() calls will fail with an authentication error.
  • Fire-and-forget: The .then() Promise is not awaited. If the dynamic import or init fails, no error is surfaced to the user. Emails will simply fail silently when triggered later.

Timing

The dynamic import is kicked off immediately when main.jsx is evaluated, which happens before React mounts. By the time a user navigates to the Dashboard and performs an approve/reject action, the EmailJS library will have loaded and initialized. However, subsequent usages of EmailJS in DashboardPage.jsx also use dynamic imports (await import('@emailjs/browser').then(m => m.default)) to ensure the module is available, making the initialization in main.jsx a warm-up optimization rather than a hard requirement.

Dual Dynamic Imports: Both main.jsx and DashboardPage.jsx use import('@emailjs/browser'). The first call loads the module and caches it; subsequent calls return the cached module instantly. The init in main.jsx ensures the library is ready by the time the Dashboard needs it, while the re-import in DashboardPage.jsx ensures the module reference is available in the local scope.
