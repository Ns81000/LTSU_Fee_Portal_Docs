User Roles

Student

Students access the system without authentication. They can submit fee requests through a two-step wizard form and track their request status using a reference number. No account or login is required.

  • Submit fee requests
  • Track request status via reference number
  • Download receipts for approved requests
  • Resubmit rejected requests
Class Representative (CR)

CRs are assigned to a single section and manage fee requests for students in that section. They log in via Supabase authentication and have a dedicated dashboard.

  • View all students in assigned section
  • Approve or reject pending fee requests
  • Batch approve multiple requests at once
  • Export data (PDF, Excel, CSV)
  • Track payment status (paid/unpaid)
Teacher

Teachers can be assigned to multiple sections and switch between them. They have the same dashboard capabilities as CRs but across multiple sections.

  • Switch between assigned sections
  • All CR capabilities across multiple sections
  • Section switcher with branch/semester labels
Admin

The admin has full system control through a dedicated panel. Admin authentication uses a separate flow with session-based validation against an environment variable.

  • Manage branches, specialisations, semesters, sections
  • Manage batch configurations and fee amounts
  • Create, edit, and deactivate staff accounts
  • Upload student data via CSV
  • View activity logs

Key Features

Fee Submission Wizard

Two-step form with roll number lookup, duplicate detection, and auto-calculated fee amounts from batch configuration.

Staff Dashboard

Comprehensive dashboard with stats, tabs, search, sort, batch operations, and multi-format data export.

Admin Panel

Nine management modules covering the entire system configuration, from branch setup to staff account provisioning.

Email Notifications

Automated approval and rejection emails via EmailJS with branded HTML templates and transaction details.

Database Schema

Ten relational tables with Row-Level Security, foreign key constraints, and server-side RPC functions.

Security

Multi-layered security with RLS policies, CSP headers, input sanitization, rate limiting, and RBAC enforcement.

Application Flow

Fee Request Lifecycle

Student Submits
→
Request Created (Pending)
→
CR/Teacher Reviews
→
Approved / Rejected
→
Email Notification Sent

Each fee request moves through a clear lifecycle. Students submit via the public form, the request enters a pending state, staff (CR or Teacher) reviews and approves or rejects the request with an optimistic concurrency check, and an email notification is dispatched to the student.

Quick Start

Prerequisites

  • Node.js 18+ installed
  • pnpm package manager
  • Supabase project (with tables, RLS, and Edge Functions configured)
  • EmailJS account with templates

Installation

# Clone the repository
git clone https://github.com/Ns81000/LTSU_Fee_Portal.git
cd LTSU_Fee_Portal

# Install dependencies
pnpm install

# Create environment file
cp .env.example .env

# Start development server
pnpm run dev

Environment Variables

VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-anon-key
VITE_EMAILJS_SERVICE_ID=your-service-id
VITE_EMAILJS_APPROVAL_TEMPLATE_ID=your-approval-template
VITE_EMAILJS_REJECTION_TEMPLATE_ID=your-rejection-template
VITE_EMAILJS_PUBLIC_KEY=your-public-key
VITE_ADMIN_EMAIL=admin@ltsu.edu.in
