Overview

The student-facing side of the LTSU Fee Portal consists of two primary pages: StudentFormPage.jsx (fee submission) and TrackingPage.jsx (request tracking). Both pages are publicly accessible and require no authentication. They are reached from the LandingPage.jsx component, which serves as the application entry point at the / route.

The Landing Page presents three GlassCard navigation options:

Public Access: Neither page uses the ProtectedRoute wrapper. Any visitor can submit a fee request or track an existing one without logging in. The only guard against misuse is roll number validation (the student must exist in the students table) and rate limiting on tracking lookups.

Fee Submission Wizard (StudentFormPage.jsx)

StudentFormPage.jsx implements a two-step wizard pattern for fee submission. The component is approximately 410 lines of React code and manages its own local state without relying on global auth context. It uses the DataContext for reference data (active batches, branches, specialisations, sections) and communicates with Supabase directly for student lookup, duplicate checking, and submission.

Roll Number Lookup Process (handleRollBlur)

When the roll number field loses focus, the following sequence executes:

  1. The raw input is sanitized via sanitizeRollNumber(value) -- strips whitespace, converts to uppercase, removes non-alphanumeric characters.
  2. If the cleaned value is empty or matches lastRollLookup.current, the function exits early (no redundant queries).
  3. rollStatus is set to 'loading', showing a spinner on the input.
  4. A Supabase query is executed: 
    supabase
  .from('students')
  .select('id, full_name, roll_number, branch_id, specialisation_id, section_id, batch_year')
  .eq('roll_number', cleaned)
  .single()
  5. If data is returned: studentData is set, rollStatus becomes 'found', and lastRollLookup is updated.
  6. If data is null: studentData is cleared, rollStatus becomes 'not_found'.
  7. If the query throws an exception: rollStatus becomes 'error'.

Auto-Filled Fields (Derived from studentData)

Once rollStatus === 'found', the following fields are computed from studentData and displayed as read-only detail rows inside a gold-tinted container:

Fee Amount is Read-Only: Students cannot manually enter or modify the fee amount. It is automatically determined by the batch configuration set by an administrator. If no batch config exists for the student's batch year, the fee amount will be null and submission will be blocked with the message "Fee amount not configured for your batch. Contact admin."

Duplicate Detection

Before advancing from Step 1 to Step 2, the form performs a server-side duplicate check to prevent students from creating multiple requests for the same semester. This is handled in the handleNext() function.

Process Flow

  1. Step 1 fields are validated: email is checked with isValidEmail(), roll number must have rollStatus === 'found'.
  2. checkingDuplicate state is set to true (shows loading on the Next button).
  3. The RPC is called: 
    supabase.rpc('check_duplicate_fee_request', {
  p_student_id: studentData.id,
  p_section_id: studentData.section_id
})
  4. The RPC returns a text value indicating the status of any existing request for this student and section combination.

Response Handling

RPC Return ValueBehaviorUI
'pending'Blocks advancement to Step 2Amber warning box: "Your fee request for this semester is already submitted and is pending review." with a link to the Track Request page
'approved'Blocks advancement to Step 2Amber info box: "Your fee request for this semester has already been approved." with a link to the Track Request page
'rejected'Allows advancement to Step 2No warning; student may resubmit after a previous rejection
'none'Allows advancement to Step 2No warning; first-time submission for this semester
Resubmission After Rejection: If a student's previous request was rejected, the duplicate check returns 'rejected' and the form permits a new submission. This allows students to correct any issues and try again without requiring admin intervention.

Submission Process

The handleSubmit() function is called when the student clicks "Submit Request" on Step 2. It performs validation, rate-limit enforcement, RPC invocation, activity logging, and transitions the UI to a success screen.

Pre-Submission Checks

  1. Fee Amount Validation: If feeAmount is null or undefined, submission is blocked with the toast message "Fee amount not configured for your batch. Contact admin."
  2. Rate Limiting: The current timestamp is compared against lastSubmitTime.current. If fewer than 10,000 milliseconds (10 seconds) have elapsed since the last submission, the toast "Please wait before submitting again." is shown and submission is aborted.

RPC Call: submit_fee_request

supabase.rpc('submit_fee_request', {
  p_student_id:    studentData.id,          // uuid
  p_roll_number:   sanitizeRollNumber(rollNumber), // text
  p_student_email: email.trim(),            // text
  p_section_id:    studentData.section_id,  // uuid
  p_batch_year:    Number(batchYear),       // integer
  p_fee_amount:    Number(feeAmount)        // numeric
})

RPC Parameters

ParameterTypeSourceDescription
p_student_iduuidstudentData.idPrimary key of the student record from the students table
p_roll_numbertextsanitizeRollNumber(rollNumber)Cleaned roll number (uppercase, alphanumeric only)
p_student_emailtextemail.trim()Student's email address for notification delivery
p_section_iduuidstudentData.section_idSection the student belongs to (determines which CR/Teacher reviews)
p_batch_yearintegerNumber(batchYear)Admission batch year of the student
p_fee_amountnumericNumber(feeAmount)Fee amount from the batch configuration

RPC Return Value

On success, the RPC returns a reference number as a text string. The format is:

LTSU-YYYY-XXXXX

Where YYYY is the current calendar year and XXXXX is a zero-padded sequential counter (e.g., LTSU-2026-00042).

Reference Number Validation

Before making any RPC call, the reference number input is validated client-side.

Validation Steps

  1. Cooldown Check: If trackCooldown is true, the error "Too many lookups. Please wait 30 seconds." is displayed and the function returns immediately.
  2. Empty Check: The input is trimmed and converted to uppercase. If the result is empty, the error "Please enter a reference number" is shown.
  3. Format Validation: The trimmed value is tested against the regex: 
    /^LTSU-\d{4}-\d{5}$/
    If it does not match, the error "Invalid format. Expected: LTSU-YYYY-XXXXX" is shown.

Valid Format Specification

SegmentPatternDescription
PrefixLTSUFixed literal string identifying the portal
Separator-Hyphen delimiter
Year\d{4}Four-digit year (e.g., 2026)
Separator-Hyphen delimiter
Sequence\d{5}Five-digit zero-padded sequential number (e.g., 00042)

Example valid reference numbers: LTSU-2026-00001, LTSU-2025-12345, LTSU-2026-00042.

Lookup Process

After passing client-side validation, the handleTrack() function proceeds with the lookup.

Rate Limiting

The tracking page implements client-side rate limiting to prevent abuse:

ParameterValueDescription
Max lookups5Maximum number of lookups before cooldown activates
Cooldown duration30 secondsTime the user must wait after hitting the limit
Counter resetAfter cooldowntrackCount resets to 0 and trackCooldown resets to false via setTimeout

Each call to handleTrack() increments trackCount. When trackCount reaches 5, the trackCooldown flag is set to true and a 30-second setTimeout is scheduled to reset both values.

RPC Call: track_fee_request

supabase.rpc('track_fee_request', {
  p_reference_number: trimmed  // uppercase, trimmed reference number
})

The RPC returns an array. If the array is empty or null, the error "No request found with this reference number" is displayed. Otherwise, the first element (data[0]) is stored in the result state.

Receipt Download

When a tracked request has the status approved, a "Download Receipt (PDF)" button is rendered below the result details. This feature uses the generateReceipt.js utility to create a full A4-sized PDF document.

Dynamic Import

The generateReceipt function is loaded via dynamic import() to avoid bundling the jsPDF library on initial page load:

const { generateReceipt } = await import('../utils/generateReceipt')

Inside generateReceipt.js, jsPDF itself is also dynamically imported:

const { default: jsPDF } = await import('jspdf')

File Download

The generated jsPDF document object is returned from generateReceipt() and saved with the filename:

LTSU_Receipt_[REFERENCE_NUMBER].pdf

For example: LTSU_Receipt_LTSU-2026-00042.pdf. The doc.save() method triggers a browser download.

Error Handling: If the dynamic import or PDF generation fails, the error is caught and logged to the console via console.error('Failed to generate receipt:', err). No user-facing error toast is displayed for receipt generation failures.

The complete data flow for a fee submission, from initial student interaction through to the success screen:

Student enters roll number in the input field
On blur: sanitizeRollNumber() cleans the input

If found: auto-fill Name, Branch, Specialisation, Section, Semester, Batch Year
Fee amount derived from activeBatches (DataContext) matching batch_year
Student enters email address
Student clicks "Next" -- triggers validateStep1()

If 'none' or 'rejected': advance to Step 2
Student reviews summary and clicks "Submit Request"

RPC returns reference number (LTSU-YYYY-XXXXX)

Error Recovery Patterns: For roll number lookup failures, the student can simply re-focus and blur the field to retry. For RPC errors during duplicate checking or submission, the student can click the button again. The tracking page's rate limiter automatically resets after 30 seconds. These patterns ensure that transient network issues do not permanently block the student from completing their task.
