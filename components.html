Component Overview

The LTSU Fee Portal organises all reusable React components inside the src/components/ directory, subdivided into four categories: UI primitives (ui/), layout shells (layout/), form helpers (forms/), and dashboard-specific widgets (dashboard/). A single top-level component, ErrorBoundary.jsx, lives at the root of src/components/.

Button Component

File: src/components/ui/Button.jsx

The Button component is the primary interactive element used across the entire application. It supports five visual variants, an integrated loading state with an animated spinner, and forwards all additional props to the underlying <button> element. The component is used for form submissions, modal actions, navigation triggers, export operations, and destructive confirmations.

Props

Prop Type Default Description
variant 'primary' | 'secondary' | 'danger' | 'ghost' | 'success' 'primary' Determines the visual style of the button. Maps to a CSS class name.
loading boolean false When true, displays an animated Loader2 spinner icon (16px) before the children content and disables the button to prevent duplicate submissions.
disabled boolean false Disables the button. The button is also automatically disabled when loading is true (via disabled={disabled || loading}).
className string '' Additional CSS classes appended to the computed variant class string.
children React.ReactNode -- The button label content. Can include text, icons, or any React nodes.
...props any -- All remaining props are spread directly onto the native <button> element. This includes onClick, type, style, aria-* attributes, and any other valid button props.

Variant Class Mapping

The component maintains an internal variants object that maps each variant name to its corresponding CSS class. If an invalid variant string is supplied, the component falls back to btn-primary.

Variant CSS Class Visual Description
'primary' btn-primary Navy background (--color-navy) with gold text (--color-gold). The default action button used for form submissions and primary actions.
'secondary' btn-secondary Transparent background with a 1px border. Used for cancel actions and secondary operations.
'danger' btn-danger Red background (--color-error) with white text. Used for destructive actions such as deletions and rejections.
'ghost' btn-ghost Fully transparent with no border. Text inherits colour from context. Used for inline actions and toolbar buttons like export triggers.
'success' btn-success Green background (--color-success) with white text. Used for approval actions.

Loading State Behaviour

When loading={true}, three things happen simultaneously:

  1. The Loader2 icon from lucide-react renders at 16px with the animate-spin CSS class, producing a smooth rotating animation.
  2. The native disabled attribute is set to true, preventing click interactions.
  3. The aria-busy attribute is set to true (or undefined when not loading), signalling the busy state to screen readers.

The children content remains visible alongside the spinner -- the spinner is prepended, not a replacement.

Accessibility

The default type attribute is "button" (not "submit"), which prevents unintended form submissions when the button is used outside of form contexts. The aria-busy attribute is conditionally applied during loading states. Consumers can override the type or add additional ARIA attributes via the rest spread.

Usage Examples

// Primary form submission
<Button onClick={handleSubmit} loading={isSubmitting}>
  Submit Request
</Button>

// Danger action with custom style
<Button variant="danger" onClick={handleDelete}>
  Delete Student
</Button>

// Ghost button for toolbar
<Button variant="ghost" onClick={() => handleExport('pdf')}>
  <Download size={14} /> PDF
</Button>

// Success approval
<Button variant="success" loading={approving} onClick={handleApprove}>
  Approve
</Button>

Input Component

File: src/components/ui/Input.jsx

The Input component is an enhanced form input that wraps the native HTML <input> element with label rendering, validation feedback (error and success states), password visibility toggle, loading indicator, and accessible ARIA bindings. It is the sole text input component used across all forms in the application -- login, student fee submission, and admin management screens.

Props

Prop Type Default Description
label string undefined When provided, renders a <label> element above the input, associated via htmlFor pointing to the input's id.
error string undefined Error message string. When truthy, applies the error CSS class to the input (red border), renders an X icon inside the input, and displays the error text below the input in a <p role="alert"> element.
success string undefined Success message string. When truthy, applies the success CSS class (green border), renders a Check icon inside the input, and displays the success text below.
loading boolean undefined Aliased internally as inputLoading. When true, displays a spinning Loader2 icon (16px) on the right side of the input. Takes visual priority over success/error icons.
type string 'text' The HTML input type. When set to 'password', an eye/eye-off toggle button is rendered to switch between masked and plain text display.
className string '' Additional CSS classes applied to the outer wrapper <div>.
id string auto-generated via useId() Explicit ID for the input element. If not provided, React's useId() hook generates a unique ID for label association and ARIA bindings.
...props any -- Spread onto the native <input> element. Covers value, onChange, placeholder, name, required, disabled, maxLength, etc.

Password Toggle

When type="password" is passed, the component enters password mode. A local state variable showPassword (managed via useState(false)) controls whether the actual input type is 'password' or 'text'. A toggle button is rendered on the right side of the input:

  • When showPassword is false: the Eye icon is shown, and the input type is 'password' (dots/bullets).
  • When showPassword is true: the EyeOff icon is shown, and the input type is 'text' (visible characters).

The toggle button has type="button" to prevent form submissions, and includes an aria-label that updates dynamically ("Hide password" or "Show password").

Validation State Rendering

The input supports three mutually exclusive visual states, determined by the precedence: error takes priority over success.

State CSS Class on Input Icon Displayed Text Below ARIA
Error error X icon (red, 16px) Error message in .input-error aria-invalid="true", aria-describedby="{id}-error"
Success success Check icon (green, 16px) Success message in .input-success aria-describedby="{id}-success"
Loading (none) Loader2 spinner (muted, 16px) (none) (none)
Default (none) (none) (none) (none)

The loading indicator takes rendering priority: when inputLoading is true, neither the success Check icon nor the error X icon is rendered. However, the error/success text below the input and the border colour still apply regardless of loading state.

Icon Positioning

Status icons and the password toggle are absolutely positioned inside the input's container div, anchored to the right side with right: 12px and vertically centred via top: 50%; transform: translateY(-50%). Multiple icons (e.g., loading spinner + password toggle) are laid out horizontally with a 4px gap using flexbox.

ID Generation

The component uses React's useId() hook to auto-generate a unique identifier when no explicit id prop is provided. This ID is used in three places: the id attribute on the input, the htmlFor attribute on the label, and the aria-describedby attribute linking to error/success message paragraphs. Each message paragraph gets a derived ID: {inputId}-error or {inputId}-success.

Usage Examples

// Basic text input with label
<Input
  label="Full Name"
  value={name}
  onChange={e => setName(e.target.value)}
  placeholder="Enter your full name"
/>

// Password input with error
<Input
  label="Password"
  type="password"
  value={password}
  onChange={e => setPassword(e.target.value)}
  error={errors.password}
/>

// Input with success feedback
<Input
  label="Staff ID"
  value={staffId}
  onChange={e => setStaffId(e.target.value)}
  success="Staff ID verified"
/>

// Loading state during async validation
<Input
  label="Roll Number"
  value={roll}
  onChange={e => setRoll(e.target.value)}
  loading={isValidating}
/>

Modal Component

File: src/components/ui/Modal.jsx

The Modal component implements an accessible dialog overlay used throughout the application for confirmation prompts, rejection reason entry, and informational displays. It renders directly into the normal React tree (not a portal), manages body scroll locking, keyboard dismissal, and focus management. The modal is the foundation upon which the RejectModal component is built.

Props

Prop Type Default Description
isOpen boolean -- Controls visibility. When false, the component returns null (renders nothing). When true, the overlay and dialog are rendered.
onClose function -- Callback invoked when the user dismisses the modal -- via backdrop click, Escape key, or the close button.
title string -- The modal heading text, rendered as an <h3> with id="modal-title" and referenced by aria-labelledby on the dialog.
children React.ReactNode -- The dialog body content. Receives any React elements -- forms, text, buttons, etc.

Scroll Locking

When the modal opens (isOpen transitions to true), a useEffect sets document.body.style.overflow = 'hidden', preventing the page behind the overlay from scrolling. When the modal closes or unmounts, the effect cleanup restores document.body.style.overflow = ''. This prevents the common issue of background content scrolling while the modal is open on mobile devices.

Keyboard Interaction

A second useEffect registers a keydown event listener on the document object. When the Escape key is pressed, onClose() is called. The listener is only active while isOpen is true and is cleaned up when the modal closes or unmounts. The effect depends on both isOpen and onClose to ensure the latest callback reference is used.

Focus Management

The Modal implements focus management via a useRef (previousFocusRef) and a third useEffect:

  1. On open: The currently focused element is saved to previousFocusRef.current. Then, using requestAnimationFrame, the component queries the modal container for all focusable elements (button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])) and focuses the first one found.
  2. On close: Focus is restored to the previously saved element via previousFocusRef.current.focus(), and the ref is cleared.

Backdrop Behaviour

The overlay div covers the entire viewport with position: fixed; inset: 0. Its background uses the navy colour at 50% opacity (rgba(11, 27, 62, 0.5)) with backdrop-filter: blur(4px) for a frosted glass effect consistent with the application's design language. Clicking anywhere on the overlay triggers onClose. The inner dialog container uses e.stopPropagation() on its onClick handler to prevent clicks inside the dialog from bubbling up and closing the modal.

Dialog Structure

The dialog itself uses the glass-card fade-in CSS class combination with a maximum width of 480px, full width, and explicit white background. The header row is a flex container with the title on the left and the close button on the right. The close button renders the X icon from lucide-react (20px) with a minimum touch target of 44x44px for mobile accessibility compliance.

ARIA Attributes

Attribute Element Value
role="dialog" Overlay container Identifies the element as a dialog
aria-modal="true" Overlay container Indicates the dialog is modal (content behind is inert)
aria-labelledby="modal-title" Overlay container Associates the dialog with its heading for screen readers
aria-label="Close" Close button Accessible name for the icon-only close button

Usage Example

const [showModal, setShowModal] = useState(false)

<Modal isOpen={showModal} onClose={() => setShowModal(false)} title="Confirm Action">
  <p>Are you sure you want to proceed?</p>
  <div style={{ display: 'flex', gap: 12, justifyContent: 'flex-end' }}>
    <Button variant="ghost" onClick={() => setShowModal(false)}>Cancel</Button>
    <Button variant="primary" onClick={handleConfirm}>Confirm</Button>
  </div>
</Modal>

Select Component

File: src/components/ui/Select.jsx

The Select component wraps a native HTML <select> element with consistent styling, label rendering, error feedback, and accessible ARIA bindings. It uses the same visual language as the Input component -- the shared input-field CSS class, the input-label class for labels, and the input-error class for validation messages. The component is used throughout admin management screens for branch, specialisation, semester, section, and batch selection.

Props

Prop Type Default Description
label string undefined When provided, renders a <label> element above the select with the input-label class, linked via htmlFor.
error string undefined Error message. When truthy, adds the error CSS class to the select and renders the error text below in a <p role="alert">.
options Array<{ value: string, label: string }> [] Array of option objects. Each is rendered as an <option> element with key={opt.value}.
placeholder string 'Select...' Text for the first disabled option element, which has value="". Serves as the default prompt when no selection has been made.
className string '' Additional CSS classes applied to the outer wrapper <div>.
id string auto-generated via useId() Explicit ID for the select element. Auto-generated if not provided.
...props any -- Spread onto the native <select> element. Covers value, onChange, disabled, name, required, etc.

ID Generation

Identical pattern to the Input component. Uses React's useId() hook to auto-generate a unique identifier, which is used for the id on the select, htmlFor on the label, and aria-describedby pointing to {selectId}-error when an error is present.

Option Rendering

The first option is always the placeholder, rendered with value="" and the placeholder text. It is not explicitly disabled in the current implementation, which means the user can select it to "clear" their selection (the empty string value is typically validated against). The options array is null-safely wrapped in (options || []) before mapping.

Styling

The select uses the shared input-field CSS class, which provides consistent height, padding, border radius, font, and colour with the Input component. The browser's native select dropdown is used (no custom dropdown overlay), styled with the project's design system variables. The error state adds a red border via the error class.

Usage Example

<Select
  label="Branch"
  value={selectedBranch}
  onChange={e => setSelectedBranch(e.target.value)}
  options={branches.map(b => ({ value: b.id, label: b.name }))}
  placeholder="Choose a branch..."
  error={errors.branch}
/>

Badge Component

File: src/components/ui/Badge.jsx

The Badge component renders a small coloured pill indicator used to display the status of fee requests. It is a minimal component that maps status strings to CSS class names, capitalises the first letter for display, and renders a <span>.

Props

Prop Type Default Description
status 'pending' | 'approved' | 'rejected' | string undefined The status string that determines both the visual variant and the displayed text. The first letter is capitalised automatically.

Style Mapping

The component uses a badgeStyles object defined at module scope:

Status CSS Classes Visual Description
'pending' badge badge-pending Yellow/amber background with dark text -- indicates a request awaiting review.
'approved' badge badge-approved Green background with dark text -- indicates an accepted request.
'rejected' badge badge-rejected Red background with dark text -- indicates a declined request.
Any other / undefined badge Base badge style with neutral/grey appearance. Fallback for unknown status values.

Text Capitalisation

The display text is computed as status.charAt(0).toUpperCase() + status.slice(1). If the status prop is falsy (undefined, null, or empty string), an empty string is rendered, producing an invisible badge element.

Usage Context

The Badge component appears in several locations throughout the application:

  • Dashboard fee request tables -- each row shows the request status as a badge.
  • Student tracking page -- the latest request status is displayed as a badge.
  • Admin student manager -- active/inactive status indicators.

Usage Example

<Badge status="pending" />   // Renders: "Pending" with amber style
<Badge status="approved" />  // Renders: "Approved" with green style
<Badge status="rejected" />  // Renders: "Rejected" with red style

GlassCard Component

File: src/components/ui/GlassCard.jsx

The GlassCard component is the primary container element in the LTSU Fee Portal's design system. It implements the application's signature frosted glass (glassmorphism) aesthetic -- a semi-transparent white background with a blur-filtered backdrop, subtle border, and rounded corners. Nearly every content section in the application is wrapped in a GlassCard.

Props

Prop Type Default Description
children React.ReactNode -- The content rendered inside the glass container.
className string '' Additional CSS classes appended to the base class list.
hover boolean true Controls hover animation. When true (default), the card has a subtle lift and shadow transition on mouse hover. When false, the no-hover class is applied, disabling the hover effect.
...props any -- All remaining props are spread onto the <div>. The style prop is specially handled: it is destructured and merged with the default padding: 24 inline style, allowing consumers to extend or override styles.

CSS Class Construction

The final class string is built as: glass-card {hover ? '' : 'no-hover'} {className}. The glass-card base class (defined in src/index.css) provides:

  • Background: Semi-transparent white with varying opacity depending on the theme context.
  • Backdrop filter: backdrop-filter: blur() for the frosted glass effect.
  • Border: Subtle semi-transparent white border.
  • Border radius: Rounded corners consistent with the design system.
  • Shadow: Soft box shadow for depth.
  • Hover transition: Subtle translateY and shadow change (disabled by no-hover class).

Style Merging

The component destructures style from the rest props and merges it with a default padding: 24 inline style using the spread operator: style={{ padding: 24, ...inlineStyle }}. This means consumers can override the padding or add any additional inline styles, and the consumer's values take precedence due to spread order.

Usage Context

The GlassCard is used as the container for virtually every content block:

  • StatsCard: Each statistics display is a GlassCard with a coloured left border.
  • Dashboard sections: Fee request tables, filters, and action areas are wrapped in GlassCards.
  • Admin panels: Each management module's content area uses GlassCard.
  • Login form: The authentication form container.
  • Student form: Each step of the multi-step form is a GlassCard.

Usage Example

// Default with hover
<GlassCard>
  <h3>Fee Request Details</h3>
  <p>Content here...</p>
</GlassCard>

// Without hover, custom padding
<GlassCard hover={false} style={{ padding: 16 }}>
  <StatsGrid />
</GlassCard>

// With additional class
<GlassCard className="my-custom-card">
  <FormContent />
</GlassCard>

Spinner Component

File: src/components/ui/Spinner.jsx

The Spinner component renders a full-page centred loading indicator used as the primary loading state throughout the application. It uses the Loader2 icon from lucide-react with the animate-spin CSS animation and includes a visually hidden text label for screen reader accessibility.

Props

Prop Type Default Description
size number 24 The pixel size passed to the Loader2 icon's size prop, controlling both width and height of the SVG icon.
className string '' Additional CSS classes appended to the Loader2 icon element alongside animate-spin.

Layout

The spinner is wrapped in a <div> with flexbox centring (display: flex; justify-content: center; align-items: center), padding: 40px, and min-height: 60vh. This ensures the spinner is vertically and horizontally centred within a significant portion of the viewport, making it suitable as a full-page loading indicator.

Accessibility

The wrapper div has role="status" and aria-label="Loading", which announces the loading state to screen readers. Additionally, a visually hidden <span> with the text "Loading..." is included using the standard CSS clip pattern (position: absolute; width: 1px; height: 1px; clip: rect(0,0,0,0); overflow: hidden). This provides redundant accessibility for assistive technologies that may not support aria-label on non-interactive elements.

Colour

The Loader2 icon uses color: var(--color-gold), rendering the spinner in the application's gold accent colour. This maintains visual consistency with the navy-and-gold design system.

Usage Context

  • ProtectedRoute: Shown while the authentication state is being resolved (if (loading) return <Spinner />).
  • DashboardPage: Displayed while fee request data is being fetched from Supabase.
  • AdminPage: Shown during initial data loading for each management module.

Usage Example

// Full-page loading
if (loading) return <Spinner />

// Larger spinner
<Spinner size={48} />

// With additional class
<Spinner className="my-spinner" />

StatsCard Component

File: src/components/ui/StatsCard.jsx

The StatsCard component displays a single numeric statistic with a label, icon, and accent colour. It uses the glass-card CSS class directly (not the GlassCard component) and features a coloured left border that visually categorises each statistic. StatsCards are used in grids on both the Dashboard and Admin Overview pages.

Props

Prop Type Default Description
icon React.ComponentType (Lucide icon) undefined A Lucide icon component (not an instance). Rendered at 18px size with the borderColor as its colour. Aliased as Icon via destructuring for JSX usage.
label string -- Descriptive text rendered below the value in --text-xs font size with muted colour. Examples: "Total Students", "Pending Requests".
value string | number -- The primary numeric value displayed prominently in --text-xl with font-weight: 700. Styled with text-overflow: ellipsis for long values.
borderColor string (CSS colour) 'var(--color-pending)' CSS colour value applied to both the 3px left border of the card and the icon colour. Used to visually distinguish different statistics.

Layout Structure

The card renders as a vertical flex column (flex-direction: column) with a 6px gap between elements:

  1. Icon (if provided) -- Lucide icon component rendered at 18px, coloured with borderColor, with flex-shrink: 0.
  2. Value -- Large bold text using the font-display class (Playfair Display). Has overflow: hidden; text-overflow: ellipsis; white-space: nowrap to handle long numbers gracefully.
  3. Label -- Small muted text in --text-xs size with white-space: nowrap.

The outer container uses glass-card class with 16px padding and a 3px left border in the accent colour.

Dashboard Usage

On the Dashboard page, four StatsCards are arranged in a CSS grid:

Label Icon Border Colour Value Source
Total Students Users var(--color-navy) Total count of students in the section
Pending Clock var(--color-pending) Count of requests with status "pending"
Approved CheckCircle var(--color-success) Count of requests with status "approved"
Rejected XCircle var(--color-error) Count of requests with status "rejected"

Tabs Component

File: src/components/ui/Tabs.jsx

The Tabs component renders a horizontal tab bar with an underline indicator on the active tab. Each tab can optionally display a count badge, making it suitable for showing filtered data counts. The component is used on the Dashboard page to switch between "All Students", "Pending", "Approved", and "Rejected" views.

Props

Prop Type Default Description
tabs Array<{ id: string, label: string, count?: number }> -- Array of tab definitions. Each tab must have a unique id and a display label. The optional count property renders a numeric badge next to the label.
active string -- The id of the currently active tab. Determines which tab receives the active visual state (gold underline, navy text, bold weight).
onChange (tabId: string) => void -- Callback invoked with the tab's id when the user clicks a tab.

Active vs Inactive Styling

Property Active Tab Inactive Tab
Bottom border 2px solid var(--color-gold) 2px solid transparent
Text colour var(--color-navy) var(--color-text-muted)
Font weight 600 400
Count badge background var(--color-navy) var(--color-border-light)
Count badge text var(--color-gold) var(--color-text-muted)

Count Badge

When a tab object includes a count property (checked via tab.count !== undefined), a small pill badge is rendered to the right of the label with border-radius: 10px, padding: 2px 8px, and font-size: var(--text-xs). The badge colours change based on whether the tab is active (navy background with gold text) or inactive (light border background with muted text).

Responsive Behaviour

The tab container uses overflow-x: auto with -webkit-overflow-scrolling: touch for smooth horizontal scrolling on iOS devices. Each tab button has flex-shrink: 0 and white-space: nowrap to prevent text wrapping, ensuring tabs remain readable when they overflow horizontally. The container has the hide-scrollbar CSS class to hide the scrollbar while maintaining scroll functionality.

Transitions

All state changes are animated with transition: color 0.2s ease, font-weight 0.2s ease, border-color 0.2s ease, producing a subtle visual shift when switching between tabs.

Usage Example

const [activeTab, setActiveTab] = useState('all')

const tabList = [
  { id: 'all', label: 'All Students', count: 120 },
  { id: 'pending', label: 'Pending', count: 15 },
  { id: 'approved', label: 'Approved', count: 98 },
  { id: 'rejected', label: 'Rejected', count: 7 },
]

<Tabs tabs={tabList} active={activeTab} onChange={setActiveTab} />

ErrorBoundary Component

File: src/components/ErrorBoundary.jsx

The ErrorBoundary is a React class component that catches JavaScript errors anywhere in its child component tree, logs the error, and renders a fallback UI instead of crashing the entire application. React error boundaries must be class components -- functional components cannot use getDerivedStateFromError or componentDidCatch.

State

State Key Type Initial Value Description
hasError boolean false Flag indicating whether an error has been caught. Drives the rendering decision between fallback UI and children.
error Error | null null The caught Error object. Currently stored but not displayed to the user in the fallback UI (only a generic message is shown).

Lifecycle Methods

static getDerivedStateFromError(error)

Called during the render phase when a descendant component throws an error. Returns a new state object { hasError: true, error }. This is a static method -- it does not have access to the component instance.

componentDidCatch(error, errorInfo)

Called during the commit phase after an error has been thrown. Currently logs the error and component stack trace to the console via console.error('ErrorBoundary caught:', error, errorInfo). The logging is wrapped in a typeof console !== 'undefined' guard for safety. This method is where error reporting service integration (e.g., Sentry, LogRocket) would be added in production.

Fallback UI

When hasError is true, the component renders a full-viewport centred error display:

  • Container: Flexbox column centred vertically and horizontally, min-height: 100vh, 40px padding, using the application's background colour (--color-bg or fallback #FAF6EE).
  • Heading: "Something went wrong" in 24px bold text with navy colour.
  • Description: "An unexpected error occurred. Please refresh the page or try again later." in 14px muted text, centred, with a max-width of 400px.
  • Action button: "Refresh Page" button styled with navy background and gold text. On click, calls window.location.reload() to perform a full page reload, which clears the error state by re-mounting the entire component tree.
Recovery Strategy: The fallback UI uses window.location.reload() rather than resetting component state. This is intentional -- a full reload ensures all contexts, providers, and Supabase authentication state are re-initialised from scratch, avoiding potential cascading errors from partially corrupted application state.

Mount Point

The ErrorBoundary wraps the entire application at the top level in src/main.jsx:

import ErrorBoundary from './components/ErrorBoundary'
import App from './App'

createRoot(document.getElementById('root')).render(
  <ErrorBoundary>
    <App />
  </ErrorBoundary>
)

Limitations

  • Error boundaries do not catch errors in event handlers (use try/catch instead).
  • Error boundaries do not catch errors in asynchronous code (promises, setTimeout).
  • Error boundaries do not catch errors during server-side rendering.
  • Error boundaries do not catch errors thrown in the error boundary itself.

ProtectedRoute Component

File: src/components/layout/ProtectedRoute.jsx

The ProtectedRoute component is a route guard that wraps page components to enforce authentication and role-based access control. It reads auth state from AuthContext and either renders the children, shows a loading spinner, or redirects to the login page.

Props

Prop Type Default Description
children React.ReactNode -- The page component to render when access is granted.
allowedRoles string[] undefined Optional array of role strings (e.g., ['cr', 'teacher']). When provided, the user's profile.role must be included in this array. When omitted, any authenticated user with an active profile gains access.

Decision Flow

Auth loading?
→
Show Spinner
No user or no profile?
→
Navigate to /login
Profile is_active is false?
→
Navigate to /login
allowedRoles provided and role not in list?
→
Navigate to /login
All checks pass
→
Render children

All redirects use React Router's <Navigate to="/login" replace /> with the replace prop, which replaces the current history entry rather than pushing a new one. This prevents the user from navigating "back" to the protected page via the browser back button.

Auth Checks (in order)

  1. Loading state: If loading is true (auth state still resolving), renders the Spinner component and stops.
  2. Authentication check: If user or profile is falsy, redirects to /login.
  3. Active account check: If profile.is_active is false, the account has been deactivated by an admin. Redirects to /login.
  4. Role check: If allowedRoles is provided and profile.role is not included, the user lacks permission. Redirects to /login.
  5. Access granted: Renders {children}.

Usage in App.jsx

<Route path="/dashboard" element={
  <ProtectedRoute allowedRoles={['cr', 'teacher']}>
    <DashboardPage />
  </ProtectedRoute>
} />

Dashboard-Specific Components

These components are used exclusively within the Dashboard and related pages. They handle data export, fee request rejection, and section switching for multi-section staff members.

ExportMenu Component

File: src/components/dashboard/ExportMenu.jsx

The ExportMenu component renders a row of three ghost-style buttons for exporting fee request data in PDF, Excel, and CSV formats. It uses dynamic imports to lazy-load the export utility modules, keeping the initial bundle size small. Each export format has its own dedicated utility function in src/utils/.

Props

Prop Type Default Description
title string -- Title used in the exported document's header. Passed through to the export utilities.
columns Array<{ key: string, label: string }> -- Column definitions for the exported table. Each column has a key (property name on data objects) and a display label.
data Array<Object> -- The array of fee request objects to export. Each object's properties are mapped to columns via the key field.
exportedBy string -- Name of the exporting user. Used in the PDF export header metadata.
sectionName string -- The name of the current section. Used in the PDF export header for context.

Export Process

The handleExport(type) function manages the export lifecycle:

  1. Sets exporting state to the type string ('pdf', 'excel', or 'csv'), which updates the button text to "Exporting..." and disables all three buttons.
  2. Dynamically imports the appropriate utility module using await import().
  3. Calls the export function with the component's props.
  4. On completion (success or error), resets exporting to null in a finally block.

Export Utilities

Type Module Function Library Description
PDF src/utils/exportPDF.js exportTablePDF jsPDF + jspdf-autotable Generates an A4 PDF with LTSU header text, section/user metadata, and an auto-formatted table of fee request data.
Excel src/utils/exportExcel.js exportExcel SheetJS (xlsx) Creates a workbook with a single worksheet, maps column labels to headers, and triggers a .xlsx download.
CSV src/utils/exportCSV.js exportCSV (none -- manual) Manually generates CSV text. String values are quoted. Triggers download via a data URI with text/csv MIME type.

Button Layout

The three buttons are rendered in a flex container with gap: 6px and flex-wrap: wrap. Each button uses the ghost variant with a Download icon (14px) from lucide-react and has a minimum touch target of 44px. When one export is in progress, all three buttons are disabled (disabled={!!exporting}). The active button shows "Exporting..." text while the other two retain their format labels.

Error Handling

Export errors are caught in a try/catch block and logged to the console (console.error). The user is not shown an error message -- the button simply returns to its normal state. This is a deliberate design choice since export failures are rare and typically caused by browser download restrictions.

RejectModal Component

File: src/components/forms/RejectModal.jsx

The RejectModal is a specialised modal dialog for collecting a rejection reason when a staff member declines a fee request. It wraps the base Modal component and adds form logic: a textarea with character validation, a character counter, and confirm/cancel buttons. The title is always "Reject Fee Request".

Props

Prop Type Default Description
isOpen boolean -- Controls modal visibility — passed through to Modal.isOpen.
onClose function -- Called when the user cancels or dismisses the modal (backdrop click, Escape key, Cancel button, close icon).
studentName string -- The student's full name, displayed in bold in the modal body for confirmation context.
rollNumber string -- The student's roll number, displayed in parentheses next to the name.
onConfirm (reason: string) => void -- Callback invoked with the trimmed rejection reason string when the user clicks "Confirm Rejection" and the reason passes validation.
loading boolean -- When true, the "Confirm Rejection" button shows a loading spinner and is disabled to prevent duplicate submissions.

Internal State

State Type Initial Description
reason string '' The current value of the rejection reason textarea. Reset to empty string when the modal opens (via useEffect on isOpen) and when the modal closes (via handleClose).

Validation

The rejection reason must be at least 10 characters (after trimming whitespace) and at most 500 characters (enforced by the textarea's maxLength attribute). The "Confirm Rejection" button is disabled until the trimmed reason reaches 10 characters via disabled={reason.trim().length < 10}. A character counter below the textarea shows {reason.trim().length}/10 characters minimum.

Modal Body Structure

  1. Student info: "{studentName} ({rollNumber})" in muted small text.
  2. Textarea: Uses the shared input-field CSS class with input-label for the "Reason for rejection" label. Configured with 4 visible rows, vertical resize only, and the DM Sans font family. Placeholder text: "Please explain the reason (minimum 10 characters)..."
  3. Character counter: Muted xs text showing current character count vs minimum.
  4. Action buttons: Right-aligned flex row with Cancel (ghost variant) and Confirm Rejection (danger variant, loading-aware).

State Reset

The textarea is reset in two places to prevent stale text from appearing:

  • A useEffect watches isOpen -- when the modal opens (transitions to true), the reason is set to ''.
  • The handleClose function explicitly sets reason to '' before calling onClose().

The handleConfirm function also resets the reason after calling onConfirm(reason.trim()), and includes a guard against execution while loading is true.

A hardcoded first button with the text "All Sections". When clicked, calls onChange('all'). Its active state is determined by activeSection === 'all'. This pill is always rendered, regardless of whether the sections array is empty.

Each section from the sections array is rendered as a pill button. The pill content has two parts:

The container uses overflow-x: auto with padding-bottom: 8px (to prevent scroll clipping) and margin-bottom: 16px for spacing from subsequent content.
