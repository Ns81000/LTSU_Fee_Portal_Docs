Admin Panel Overview

The Admin Panel is a self-contained administration interface accessed at the /admin route. It implements its own authentication flow, completely separate from the staff login used by CRs and Teachers. The panel is responsible for all system configuration -- from academic structures (branches, specialisations, semesters, sections) to user management (staff accounts, student records) and fee configuration (batch year pricing).

Authentication

The admin login is session-based and validates the signed-in user's email against the VITE_ADMIN_EMAIL environment variable. Only when the authenticated email matches does the panel grant access. The admin credentials are:

FieldValue
Emailadmin@ltsu.edu.in
Staff IDADMIN001

If the user signs in successfully via Supabase Auth but their email does not match VITE_ADMIN_EMAIL, the session is immediately signed out and the login fails silently (no email enumeration). After successful admin login, the action is recorded in the activity log with action type admin_login.

Branch Manager (BranchManager.jsx)

The Branch Manager provides full CRUD operations on the branches table. Branches are the top-level academic unit in the hierarchy: Branch -> Specialisation -> Section.

Database Fields

FieldTypeConstraintsDescription
nametextUNIQUE, NOT NULLFull name of the branch (e.g. "Computer Science and Engineering")
codetextUNIQUE, NOT NULLShort code (e.g. "CSE", "ECE", "ME")
is_activebooleanDEFAULT trueActive/inactive toggle
created_attimestamptzDEFAULT now()Creation timestamp

Both name and code have UNIQUE constraints enforced at the database level. Attempting to insert a duplicate will result in a Supabase error that is caught and displayed via toast.

User Interface

The page displays a header row with the title "Branches" and an "Add Branch" button. Below this is a table with columns: Name, Code, Status, Created At, and Actions. Branches are sorted by created_at descending (newest first) using useMemo.

The Status column shows a badge -- Active or Inactive -- reflecting the is_active boolean.

Actions

  • Edit: Opens a modal pre-filled with the branch's current name and code. On save, calls supabase.from('branches').update({ name, code }).eq('id', editing.id).
  • Toggle Active/Inactive: Flips the is_active boolean. Deactivated branches remain in the database but are typically filtered out of selection dropdowns throughout the application.
  • Delete: Shows a confirmation dialog warning: "Permanently delete branch [name]? This will fail if any specialisations, sections, or students still reference this branch. Consider deactivating instead to preserve historical data." If confirmed, calls supabase.from('branches').delete().eq('id', branch.id). The delete will fail at the database level if foreign key references exist.

Activity Logging

Every create, update, toggle, and delete operation logs to the activity_logs table with the following action types:

  • admin_created_branch
  • admin_updated_branch
  • admin_toggled_branch (with details indicating activated or deactivated)
  • admin_deleted_branch

All actions are recorded with performedByRole: 'admin' and performedByName: 'Administrator'. Input values are sanitised via sanitizeText() before being sent to the database.

Dependency Cascade: Deleting a branch will fail if any specialisations, sections, or students reference it via foreign keys. The recommended pattern is to deactivate rather than delete, which preserves historical data while hiding the branch from new selections.

Parent Dependency: Specialisations require an active parent branch. If a branch is deactivated, its specialisations remain in the database but the branch will no longer appear in the creation dropdown, effectively preventing new specialisations from being added under it.

Central Role: Sections are the linking entity between academic structure and users. A section ties together a branch + specialisation + semester into an assignable unit. Staff are assigned to sections via the user_sections junction table, and students have a direct section_id foreign key. This makes sections the most interconnected entity in the system.

Batch Manager (BatchManager.jsx)

The Batch Manager configures the batch_configs table, which determines the fee amount that students must pay based on their batch year. Each batch year has exactly one fee amount, and the table is the source of truth for fee calculations across the entire system.

Database Fields

is_activebooleanDEFAULT trueActive/inactive toggle
created_attimestamptzDEFAULT now()Creation timestamp

The batch_year field has a UNIQUE constraint. Client-side validation ensures the year is a valid 4-digit integer and the fee amount is a positive number. Duplicate batch year checks are performed against the existing batches array before insert.

User Interface

The page title is "Batch Years & Fees". The table displays columns: Batch Year, Fee Amount (formatted as currency using formatCurrency()), Status, Created At, and Actions. Batches are sorted by created_at descending.

Actions

  • Edit Fee: Opens a modal allowing the admin to change the fee amount for an existing batch year. The batch year itself cannot be changed once created -- only the fee amount is editable. The update calls supabase.from('batch_configs').update({ fee_amount }).eq('id', editing.id).
  • Toggle Active/Inactive: Flips is_active. Deactivated batch configs will not appear in batch year selection dropdowns.
  • Delete: Confirmation warns: "Permanently delete batch year [year] (fee: [amount])? This removes the fee configuration. Students in this batch year will no longer have a fee amount set. Consider deactivating instead to preserve historical data."

Activity Logging

Action types: admin_created_batch, admin_updated_batch, admin_toggled_batch, admin_deleted_batch. Details include batch_year and fee_amount.

Fee Impact: Deleting a batch configuration removes the fee amount for all students in that batch year. This means any new fee requests for those students will not have a configured fee amount to reference. Always prefer deactivation over deletion for batch years that have ever had students or fee requests.

Staff Manager (StaffManager.jsx)

The Staff Manager is the most complex module in the admin panel, spanning approximately 737 lines. It handles the complete lifecycle of staff accounts: creation with auto-generated credentials, section assignments, password resets, activation/deactivation, and deletion. Staff are divided into two roles: CR (Class Representative) and Teacher.

Staff Creation

Clicking the "Create CR Account" or "Create Teacher Account" button (label changes based on active tab) opens a modal with the following fields:

Auto-Generated Staff ID

The Staff ID is generated using the generateStaffId(role) function with the following algorithm:

  1. Determine prefix based on role: TCH for teachers, CR for CRs.
  2. Generate a timestamp component: take Date.now().toString(36).toUpperCase() and extract the last 4 characters.
  3. Generate a random component: create 3 random bytes via crypto.getRandomValues(new Uint8Array(3)), convert each byte modulo 36 to a base-36 uppercase character.
  4. Combine: {prefix}-{timestamp}{random} (e.g. TCH-1K5XABC or CR-2M9NDEF).
const generateStaffId = (role) => {
  const prefix = role === 'teacher' ? 'TCH' : 'CR'
  const stamp = Date.now().toString(36).toUpperCase().slice(-4)
  const rand = Array.from(crypto.getRandomValues(new Uint8Array(3)))
    .map(b => (b % 36).toString(36).toUpperCase())
    .join('')
  return `${prefix}-${stamp}${rand}`
}

Auto-Generated Password

The password is generated using the generatePassword() function, which produces a 12-character cryptographically random password with guaranteed character diversity:

  1. Define character pools: uppercase (ABCDEFGHJKLMNPQRSTUVWXYZ -- note: ambiguous characters I, O removed), lowercase (abcdefghjkmnpqrstuvwxyz -- i, l, o removed), digits (23456789 -- 0 and 1 removed), and symbols (!@#$%&*).
  2. Pick one character from each pool to guarantee at least 1 uppercase, 1 lowercase, 1 digit, and 1 special character.
  3. Fill the remaining 8 positions with random characters from the combined pool of all categories.
  4. Shuffle the entire 12-character array using the Fisher-Yates algorithm with crypto.getRandomValues as the random source.
const generatePassword = () => {
  const upper = 'ABCDEFGHJKLMNPQRSTUVWXYZ'
  const lower = 'abcdefghjkmnpqrstuvwxyz'
  const digits = '23456789'
  const symbols = '!@#$%&*'
  const all = upper + lower + digits + symbols
  const pick = (s) => s[secureRandom(s.length)]
  let pw = [pick(upper), pick(lower), pick(digits), pick(symbols)]
  for (let i = 0; i < 8; i++) pw.push(pick(all))
  // Fisher-Yates shuffle
  for (let i = pw.length - 1; i > 0; i--) {
    const j = secureRandom(i + 1);
    [pw[i], pw[j]] = [pw[j], pw[i]]
  }
  return pw.join('')
}

The secureRandom(max) helper uses crypto.getRandomValues(new Uint32Array(1)) for cryptographically secure randomness, unlike Math.random().

Section Assignment on Creation

Below the credential fields, a section assignment area is displayed, using cascading selects for Branch, Specialisation, and Section. The behaviour differs by role:

  • CR: Assigned to exactly one section. The section select works as a standard single-value dropdown. The selected section_id is sent as a single-element array.
  • Teacher: Assigned to one or more sections. After selecting a section, the admin clicks "+ Add Section" to add it to a list. Sections appear as removable tags below the selects. Each tag displays the section name, branch name, and specialisation name. Duplicate section additions show a warning toast.

Submission Flow

On submit, the component invokes the admin-actions Supabase Edge Function with the following payload:

await supabase.functions.invoke('admin-actions', {
  body: {
    action: 'create_user',
    email,
    password,
    fullName,
    role,        // 'cr' or 'teacher'
    staffId,
    sectionIds,  // array of section UUIDs
  },
})

The Edge Function performs three operations server-side:

  1. Creates a new entry in auth.users (Supabase Auth) with the provided email and password.
  2. Creates a corresponding record in the public.users table with the staff details (full_name, email, role, staff_id, is_active).
  3. Creates user_sections junction records linking the new user to their assigned sections.
Credentials Display: After successful creation, the admin should note down or share the generated staff ID and password with the new staff member. There is no way to retrieve these credentials later -- the password is only visible at the moment of creation.

Staff Actions

Edit Sections

Opens a modal titled "Edit Sections -- [staff name]". The modal displays the currently assigned sections as removable tags, followed by cascading selects (Branch -> Specialisation -> Section) and an "Add Section" button. For CRs, selecting a new section replaces the current one (single section only). For Teachers, sections accumulate in the list.

On save, the component performs two database operations:

  1. Deletes all existing user_sections records for the user: supabase.from('user_sections').delete().eq('user_id', user.id)
  2. Inserts new user_sections records for each assigned section.

This "delete all then re-insert" pattern ensures clean state without needing to diff existing vs. new assignments.

Reset Password

Opens a modal where the admin enters a new password (minimum 6 characters). A new password can also be auto-generated using the same generatePassword() function. On submit, calls the admin-actions Edge Function:

await supabase.functions.invoke('admin-actions', {
  body: {
    action: 'reset_password',
    userId: resetModal.user.id,
    newPassword,
  },
})

The Edge Function uses the Supabase Admin API (service role key) to update the user's password in auth.users.

Toggle Active/Inactive

Flips the is_active flag in the public.users table. Deactivated staff members are blocked from logging in -- the login flow in AuthContext checks is_active after authentication and rejects inactive accounts. This is a soft disable that preserves the account and all its data.

Delete Staff Account

The most destructive action. Shows a strong confirmation warning: "Permanently delete [ROLE] account [name]? This will remove their section assignments and deactivate their auth account. Any fee requests they previously reviewed will lose their reviewer reference. Consider deactivating instead to preserve audit history."

If confirmed, the deletion proceeds in three steps:

  1. Remove section assignments: supabase.from('user_sections').delete().eq('user_id', user.id)
  2. Deactivate auth account: Call Edge Function with action: 'deactivate_user' to disable the auth.users entry server-side.
  3. Delete public record: supabase.from('users').delete().eq('id', user.id)

CSV Uploader (CSVUploader.jsx)

The CSV Uploader provides a 3-step wizard interface for bulk-importing student records from CSV files. It uses the PapaParse library for client-side CSV parsing.

Step 1: Select Target

The admin selects the target academic context for the uploaded students using four cascading dropdowns:

  1. Branch -- from active branches
  2. Specialisation -- filtered by selected branch
  3. Section -- filtered by selected specialisation
  4. Batch Year -- from active batch configs

All four selections are required before the upload can proceed. These values determine the branch_id, specialisation_id, section_id, and batch_year assigned to every student in the upload.

Step 2: Upload File

A drag-and-drop zone accepts CSV files. The zone is styled with a dashed border that highlights in gold on hover. Alternatively, clicking the zone opens a native file picker. Constraints:

ConstraintLimitError Message
File sizeMaximum 1 MB"File too large ([size]MB). Maximum 1MB."
File type.csv and .txt only"Only .csv and .txt files are supported"
Row countMaximum 500 rows"Too many rows ([count]). Maximum 500 per upload."

The CSV is parsed using Papa.parse(file, { header: true, skipEmptyLines: true }). The parser looks for two required columns (case-insensitive, with multiple accepted header names):

Required ColumnAccepted Header Names
Roll NumberRoll Number, rollnumber, roll_number
NameName, full_name, Student Name

Roll numbers are sanitised via sanitizeRollNumber() and names via sanitizeText(). Rows where either value is empty after sanitisation are filtered out.

An "Download example CSV" link is provided below the upload zone, pointing to /student_list.csv in the public directory.

Step 3: Preview

After successful parsing, a preview table displays the first 10 parsed rows with columns: # (row index), Roll Number, and Name. If more than 10 rows exist, a text note reads "...and [N] more rows".

A "Confirm Upload ([count] students)" button triggers the actual insert process.

Upload Process

The upload function performs the following steps:

  1. Duplicate detection: Queries the students table for all roll numbers in the upload batch: supabase.from('students').select('roll_number').in('roll_number', rollNumbers). Roll numbers that already exist in the database are collected into a Set.
  2. Filter new records: Only rows whose roll numbers are NOT in the existing set are prepared for insertion.
  3. Insert: New student records are inserted with the target branch, specialisation, section, and batch year from Step 1, plus is_active: true.
  4. Activity logging: Logs the upload with action type admin_uploaded_students, including details: total count, added count, skipped count, and file name.

Upload Results

After completion, the wizard displays a results card with:

  • A green checkmark and "Upload Complete" header
  • Count of students added and skipped (duplicates)
  • If any were skipped, a list of the skipped roll numbers displayed as small tags
  • An "Upload More" button to reset the wizard
Duplicate Handling: The uploader checks for duplicate roll numbers against the entire students table, not just within the current upload file. Duplicates within the file itself are not detected at the preview stage -- they would cause a database unique constraint violation on insert. The system silently skips any students whose roll numbers already exist in the database.

Student Manager (StudentManager.jsx)

The Student Manager provides full CRUD operations on the students table. At approximately 560 lines, it is the second most complex admin module after the Staff Manager. It features advanced filtering, pagination, responsive mobile card layouts, and cascading form selects.

Data Fetching

Students are fetched with joined relations to resolve branch, specialisation, and section names:

const { data: studentData } = await supabase
  .from('students')
  .select('*, branches(name), specialisations(name), sections(name)')
  .order('uploaded_at', { ascending: false })

All students are loaded into local state on mount. Filtering and pagination are performed client-side on the full dataset.

Results are paginated at 20 students per page. The pagination control appears when there are more results than the page size and includes:

  • First button -- jumps to page 0
  • Previous button (chevron left) -- goes back one page
  • Numbered page buttons -- up to 5 visible on desktop, 3 on mobile. The active page is highlighted with navy background and gold text. Ellipsis ("...") markers appear when pages are truncated.
  • Next button (chevron right) -- advances one page
  • Last button -- jumps to the final page

Navigation buttons are disabled at their respective boundaries (First and Previous disabled on page 0; Next and Last disabled on the last page).

Create / Edit Student

Both create and edit share the same modal with the title "Add Student" or "Edit Student". The modal contains:

  • Roll Number -- text input, sanitised via sanitizeRollNumber()
  • Full Name -- text input, sanitised via sanitizeText()
  • Branch -- select dropdown from active branches
  • Specialisation -- cascaded from branch selection
  • Section -- cascaded from specialisation selection
  • Batch Year -- select dropdown from active batch configs

All six fields are required. When editing, the modal is pre-filled with the student's current values, including correctly initialised cascading selects.

Actions

  • Edit: Opens the modal pre-filled with the student's current data.
  • Toggle Active/Inactive: Flips is_active. Deactivated students remain in the database but will not appear in fee request forms.
  • Delete: Shows a strong confirmation warning: "WARNING: Permanently delete student [name] ([roll_number])? This will ALSO DELETE all their fee request history (approved, pending, rejected). This data CANNOT be recovered. Consider deactivating instead if you want to preserve records." If confirmed, deletes the student record entirely.

Activity Logging

Action types: admin_created_student, admin_updated_student (also used for toggle), admin_deleted_student. Details include student name and roll number.

Data Loss Warning: Deleting a student cascades to all their fee request records. This means approved, pending, and rejected fee request history is permanently destroyed. This is the most destructive operation in the Student Manager and should only be used when the student record was created in error. For students who have graduated or transferred, deactivation is always the correct approach.
Performance Note: All students are loaded into memory and filtered client-side. For institutions with thousands of students, this approach works well because the data payload is relatively small (only student records with resolved names). However, the full dataset is re-fetched from Supabase after every create, update, toggle, or delete operation.
