Authentication Overview

The LTSU Fee Portal uses Supabase Auth as its sole authentication provider, handling all credential-based authentication via email and password. There is no OAuth, magic link, or SSO integration. The system supports three authenticated roles that correspond to distinct access levels within the application.

Role Authentication Method Access Scope
admin Email + Password (verified against VITE_ADMIN_EMAIL) Full admin panel: 9 management modules + activity log
cr Email + Password (Supabase Auth) Dashboard: review/approve/reject fee requests for assigned sections
teacher Email + Password (Supabase Auth) Dashboard: review/approve/reject fee requests for assigned sections

Students do not need authentication. The student-facing pages (/submit for fee submission and /track for tracking request status) are entirely public. Students identify themselves by roll number, not by a login session.

Users Table Structure

The users table in the public schema extends auth.users with application-specific fields. When a staff account is created, a corresponding record is inserted into both auth.users (managed by Supabase) and public.users (managed by the application).

Column Type Description
id uuid (PK, FK to auth.users) Primary key, references the Supabase auth user ID
role text One of: admin, cr, teacher
full_name text Display name of the staff member
staff_id text (unique) Auto-generated identifier (e.g., TCH-17100523894a)
email text (unique) Login email address
is_active boolean Account activation flag; false prevents login
created_at timestamptz Timestamp of account creation
created_by uuid (FK) ID of the admin who created the account
Supabase Responsibilities: Supabase Auth handles JWT token generation, session refresh (automatic), secure password hashing (bcrypt), and the auth.users table. The application never stores or processes raw passwords -- all credential handling is delegated to Supabase's server-side auth engine.

Login Flow (LoginPage.jsx)

Step-by-Step Login Process

The login flow is implemented in LoginPage.jsx and involves multiple validation layers before granting access:

  1. User navigates to /login. The page renders a centered login card with a role selector and credential fields.
  2. Role selection. Two pill-shaped buttons allow the user to choose between "CR" (Class Representative) and "Teacher". The selected role is stored in local state as role ('cr' or 'teacher'). This selection determines which role the profile will be validated against after authentication.
  3. User enters email and password. Both fields are required HTML inputs. The email field uses type="email" for basic browser validation.
  4. Lockout check. Before any network request, the handler checks whether a lockout is active. If lockoutUntil is set and Date.now() < lockoutUntil, the form displays a countdown message and returns early without calling Supabase.
  5. Supabase authentication. The login(email, password) function from AuthContext is called, which internally invokes supabase.auth.signInWithPassword({ email, password }). On success, this returns the authenticated user object.
  6. Profile fetch. After successful Supabase auth, the component directly queries the users table: supabase.from('users').select('*').eq('id', user.id).single(). This retrieves the application-level profile record.
  7. Account existence check. If no profile is returned (!profile), the user sees "Account not found. Contact your administrator." and login halts.
  8. Active status check. If profile.is_active is false, the error "Your account has been deactivated. Contact the administrator." is displayed and login halts.
  9. Role verification. The profile's role field is compared against the role pill the user selected. If profile.role !== role, Supabase auth is signed out immediately (supabase.auth.signOut()) and the error "Incorrect role selected." is shown. This prevents a teacher from logging in via the CR pill and vice versa.
  10. Activity log entry. On successful validation, an activity log record is created via logAction() with actionType: 'staff_login', performedByRole: profile.role, performedByName: profile.full_name, and performedById: profile.staff_id.
  11. Redirect. A success toast is shown and the user is navigated to /dashboard using React Router's useNavigate().

Login Lockout Mechanism

A client-side brute-force protection system prevents rapid successive login attempts:

State Variable Type Purpose
attempts number Tracks the count of consecutive failed login attempts (initialized to 0)
lockoutUntil number | null Unix timestamp (ms) when the lockout expires; null when no lockout is active

Behaviour on failed login:

  1. The catch block increments attempts by 1 using a state updater function.
  2. If the new count is >= 5, lockoutUntil is set to Date.now() + 60000 (60-second cooldown).
  3. While locked out, the submit handler checks lockoutUntil at the top and displays "Too many attempts. Try again in Xs." where X is the remaining seconds.
// Lockout logic in the catch block of handleSubmit
setAttempts(prev => {
  const next = prev + 1
  if (next >= 5) setLockoutUntil(Date.now() + 60_000)
  return next
})
Client-Side Only: The lockout mechanism is implemented entirely in React state. Refreshing the page resets the counter. This provides a first line of defence against accidental rapid retries but is not a substitute for server-side rate limiting. Supabase itself applies its own rate limits on the signInWithPassword endpoint.

AuthContext (AuthContext.jsx)

Context Architecture

The AuthContext is created via createContext() and exposed through an AuthProvider wrapper component placed at the root of the component tree in App.jsx. All child components can access authentication state via the useAuth() custom hook.

State Variables

State Type Description
user object | null The Supabase auth.user object containing id, email, aud, etc.
profile object | null The application-level user record from the public.users table
sections array Array of section objects the user is assigned to, with nested semester/branch/specialisation data
loading boolean True while the initial session check and profile fetch are in progress

Refs

Ref Type Purpose
fetchingRef useRef(false) Guard to prevent duplicate fetchProfile calls. If fetchingRef.current is true, subsequent calls return immediately.

fetchProfile Function

The core data-fetching function retrieves the user's profile and their assigned sections with all related academic data:

const fetchProfile = useCallback(async (userId) => {
  if (fetchingRef.current) return
  fetchingRef.current = true
  try {
    // Step 1: Fetch the user profile
    const { data: profileData } = await supabase
      .from('users')
      .select('*')
      .eq('id', userId)
      .single()

    if (profileData) {
      setProfile(profileData)

      // Step 2: Fetch assigned sections with nested relations
      const { data: sectionData } = await supabase
        .from('user_sections')
        .select(`
          section_id,
          sections(
            id, name, branch_id, specialisation_id, semester_id,
            semesters(id, name, number),
            branches(name),
            specialisations(name)
          )
        `)
        .eq('user_id', userId)

      setSections(sectionData?.map(s => s.sections) || [])
    }
  } finally {
    fetchingRef.current = false
    setLoading(false)
  }
}, [])

This two-step fetch ensures the profile is available before sections are loaded. The fetchingRef guard prevents race conditions when onAuthStateChange fires multiple times rapidly (which can happen during session restoration on page load).

Auth State Listener

On mount, the AuthProvider registers a Supabase auth state change listener. This fires on sign-in, sign-out, and token refresh events:

useEffect(() => {
  // Check existing session on mount
  supabase.auth.getSession().then(({ data: { session } }) => {
    if (session?.user) {
      setUser(session.user)
      fetchProfile(session.user.id)
    } else {
      setLoading(false)
    }
  })

  // Listen for auth state changes
  const { data: { subscription } } = supabase.auth.onAuthStateChange(
    (_event, session) => {
      if (session?.user) {
        setUser(session.user)
        fetchProfile(session.user.id)
      } else {
        setUser(null)
        setProfile(null)
        setSections([])
        setLoading(false)
      }
    }
  )

  return () => subscription.unsubscribe()
}, [fetchProfile])

On SIGNED_IN, the user state is set and fetchProfile() is called to load application data. On SIGNED_OUT, all state is cleared to null / empty arrays.

Exposed Context Value

The context value is wrapped in useMemo to prevent unnecessary re-renders across the component tree:

const value = useMemo(() => ({
  user,       // Supabase auth.user object
  profile,    // public.users record
  sections,   // Array of assigned section objects
  loading,    // Boolean: initial load in progress
  login,      // (email, password) => Promise
  logout,     // () => Promise (signs out, clears state)
}), [user, profile, sections, loading, login, logout])

Function Signatures

Function Parameters Returns Description
login (email: string, password: string) Promise<data> Calls supabase.auth.signInWithPassword(); throws on error
logout None Promise<void> Calls supabase.auth.signOut(); clears user, profile, and sections

useAuth Hook

export const useAuth = () => {
  const context = useContext(AuthContext)
  if (!context)
    throw new Error('useAuth must be used within an AuthProvider')
  return context
}

The hook includes a runtime guard that throws if called outside the AuthProvider tree. This catches misconfiguration errors during development.

Admin Authentication (AdminPage.jsx)

Separate Auth Flow

The admin panel uses a completely separate authentication flow from the regular staff login. It does not share the /login route or the AuthContext login mechanism. Instead, AdminPage.jsx manages its own local authentication state.

Authentication Steps

  1. Session check on mount. The component calls supabase.auth.getSession() to check if an existing session belongs to the admin. If session.user.email matches VITE_ADMIN_EMAIL (case-insensitive comparison), the component sets authenticated = true and skips the login form.
  2. Login form. If no valid admin session exists, a login card is rendered with email and password fields and a shield icon.
  3. Supabase auth. On submit, supabase.auth.signInWithPassword({ email, password }) is called.
  4. Admin email verification. After successful Supabase authentication, the entered email is compared (case-insensitive) against the VITE_ADMIN_EMAIL environment variable. If they do not match, supabase.auth.signOut() is called immediately and the login fails with a generic error message.
  5. Activity log. On successful admin login, an activity log entry is created with actionType: 'admin_login', performedByRole: 'admin', performedByName: 'Administrator'.
  6. Admin dashboard renders. The AdminSidebar component and the active management module are displayed.

Security Note: The admin email check is performed client-side after Supabase authentication succeeds. This prevents unauthorized users from accessing the admin UI, but the true security boundary is enforced at the database layer via Row-Level Security policies. Even if a non-admin user bypassed the client-side check, RLS policies would prevent them from executing admin-only database operations.

Route Protection (ProtectedRoute.jsx)

Guard Component

The ProtectedRoute component wraps protected routes in App.jsx, enforcing authentication and role-based access before rendering child components. It consumes the AuthContext via useAuth().

Props

Prop Type Description
children ReactNode The component to render if all checks pass
allowedRoles string[] | undefined Array of permitted roles (e.g., ['cr', 'teacher']). If omitted, any authenticated active user is allowed.

Check Sequence

  1. If loading is true -- renders <Spinner /> (waiting for session check to complete).
  2. If !user or !profile -- redirects to /login via <Navigate to="/login" replace />.
  3. If !profile.is_active -- redirects to /login.
  4. If allowedRoles is provided and !allowedRoles.includes(profile.role) -- redirects to /login.
  5. All checks pass -- renders children.
export default function ProtectedRoute({ children, allowedRoles }) {
  const { user, profile, loading } = useAuth()

  if (loading) return <Spinner />

  if (!user || !profile) return <Navigate to="/login" replace />

  if (!profile.is_active) return <Navigate to="/login" replace />

  if (allowedRoles && !allowedRoles.includes(profile.role)) {
    return <Navigate to="/login" replace />
  }

  return children
}

Lazy Loading: All page components in App.jsx are loaded via React.lazy() with a <Suspense fallback={<Spinner />}> wrapper. This means the JavaScript bundle for each page is only downloaded when that route is first accessed. The ProtectedRoute guard runs after the lazy component resolves but before the component's own effects execute.

Session Management

Staff Session (Supabase-Managed)

Supabase Auth manages session lifecycle automatically for staff users (CR and teacher roles):

Aspect Behaviour
Storage Session tokens (access token + refresh token) are persisted in localStorage by the Supabase client library
Token format JWT (JSON Web Token) containing the user's sub (user ID), role, aud, and expiry claims
Token refresh Supabase automatically refreshes the JWT before it expires using the refresh token. No application code is needed.
State detection The onAuthStateChange listener in AuthContext detects all auth events (sign-in, sign-out, token refresh) and updates application state accordingly
Profile re-fetch fetchProfile() is called on every auth state change that includes a user, ensuring the profile data is always current
Persistence Sessions survive page refreshes and browser restarts (persisted in localStorage). Users remain logged in until they explicitly sign out or the refresh token expires.

Admin Session

The admin authentication state is managed differently from staff sessions:

  • The authenticated boolean is stored in component-level React state, not in sessionStorage.
  • On page refresh, the component re-checks the existing Supabase session by calling supabase.auth.getSession() and comparing the user's email against VITE_ADMIN_EMAIL.
  • Closing the browser tab does not invalidate the Supabase session in localStorage, but the admin component will re-verify the email on next load.
  • On logout, supabase.auth.signOut() is called, which clears the session from localStorage and resets all local component state.

Token Expiry: If the Supabase refresh token itself expires (due to prolonged inactivity beyond the server-configured refresh token lifetime), the user will be silently signed out on the next auth state change event. The onAuthStateChange listener will fire with a null session, clearing all application state and effectively redirecting the user to the login page on the next navigation attempt through ProtectedRoute.

User Account Lifecycle

Full Lifecycle

Staff accounts (CR and teacher) follow a controlled lifecycle managed entirely by the admin:

  1. Account Creation. The admin creates a staff account via the StaffManager component in the admin panel. This triggers a Supabase Edge Function (admin-actions) that:
    • Creates a record in auth.users (Supabase-managed authentication table) with the provided email and auto-generated password.
    • Creates a corresponding record in public.users with the role, full name, auto-generated staff ID, email, and is_active = true.
    • Creates entries in user_sections to link the staff member to their assigned sections.
  2. Staff ID Generation. Staff IDs are auto-generated with the following format:
    • Teachers: TCH-{timestamp}{random} (e.g., TCH-17100523894a)
    • Class Representatives: CR-{timestamp}{random} (e.g., CR-1710052389b7)
    The timestamp provides uniqueness, and random characters add collision resistance.
  3. Password Generation. An initial password of 12 characters is auto-generated using crypto.getRandomValues(). The password includes a mix of:
    • Uppercase letters (A-Z)
    • Lowercase letters (a-z)
    • Digits (0-9)
    • Special symbols (!@#$%^&*)
    The generated password is shown to the admin once during account creation. It is never stored in plaintext -- Supabase hashes it using bcrypt before persisting.
  4. First Login. The staff member signs in at /login using the admin-provided email and generated password. On successful login, the activity log records the staff_login event.
  5. Password Reset. The admin can reset a staff member's password via the StaffManager component, which calls the Edge Function. A new 12-character password is generated and displayed to the admin. The old password is invalidated immediately.
  6. Account Deactivation. The admin toggles is_active to false via the StaffManager. The effect is:
    • Existing sessions are not forcefully terminated.
    • On the next login attempt, step 8 of the login flow (active status check) fails, and the user sees "Your account has been deactivated."
    • If the user is already logged in, the ProtectedRoute guard checks profile.is_active on re-render (e.g., after profile re-fetch on auth state change). If inactive, the user is redirected to /login.
  7. Account Reactivation. The admin can toggle is_active back to true. The staff member can then log in normally with their existing credentials (unless the password was also reset).

No Self-Service: Staff members cannot create their own accounts, change their own passwords, or modify their profile information. All account management operations are restricted to the admin through the admin panel and the admin-actions Edge Function. This centralised control ensures that only authorised personnel have system access.
Password Visibility: The auto-generated password is displayed in the admin UI only at the moment of creation or reset. There is no "show password later" feature. If the admin does not record the password, a new one must be generated via the reset function. The application never stores passwords in plaintext anywhere -- not in the database, not in the browser, and not in logs.
