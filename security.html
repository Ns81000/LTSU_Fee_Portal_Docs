Security Overview

The LTSU Fee Portal implements security at multiple layers, ensuring that no single point of compromise can grant unauthorized access to the system. Each layer operates independently and reinforces the others.

1. Database Level
Row-Level Security (RLS) on all 10 tables
2. Application Level
Role-Based Access Control (RBAC), input sanitization, rate limiting
3. Network Level
Content Security Policy headers, HTTPS enforcement
4. Server Level
Edge Functions with service_role key for privileged operations
5. Audit Level
Comprehensive activity logging with activity_logs table
Defense in Depth: Even if a client-side check is bypassed (e.g., by manipulating React state), the database-level RLS policies and Edge Function authorization checks prevent unauthorized data access or modification. The security model does not rely on any single layer alone.

Row-Level Security (RLS)

All 10 database tables have RLS enabled. RLS enforces data access at the PostgreSQL level — queries only return rows the user is authorized to see. This means that even if the application code contains a bug that constructs an overly broad query, the database itself will filter the results before returning them.

RLS Bypass Mechanisms

Two mechanisms exist for bypassing RLS when required:

  • SECURITY DEFINER RPC functions: Used to grant public access to specific operations that do not require authentication, such as student fee submission (submit_fee_request) and tracking (track_fee_request). These functions execute with the privileges of the function creator, not the caller.
  • Service role key: The supabase service_role key is used exclusively within Edge Functions. This key bypasses RLS entirely and has full database access. It is never exposed to the client.

SECURITY DEFINER Functions: The submit_fee_request and track_fee_request RPCs are declared with SECURITY DEFINER. This means they execute with the privileges of the database role that created them, not the anonymous caller. These are the only entry points for unauthenticated users to interact with the database. The functions perform strict input validation internally before executing any queries.
Service Role Key Exposure: The service_role key must never be exposed in client-side code, environment variables accessible to the browser, or version control. It is stored exclusively as a Supabase Edge Function secret and used only within the admin-actions Edge Function. Leaking this key would grant unrestricted database access, bypassing all RLS policies.

Content Security Policy (CSP)

The application enforces a Content Security Policy via a <meta> tag in index.html. CSP restricts which resources the browser is allowed to load, mitigating cross-site scripting (XSS), data injection, and clickjacking attacks.

CSP Directive Breakdown

Directive Value Purpose
default-src 'self' Fallback for all resource types — only same-origin resources allowed unless overridden by a more specific directive
script-src 'self' Only scripts from the same origin are permitted. No inline scripts, no eval(), no external CDNs
style-src 'self' 'unsafe-inline' https://fonts.googleapis.com Allows same-origin stylesheets, inline styles (required for React component styling), and Google Fonts CSS
font-src https://fonts.gstatic.com Allows font file downloads from Google Fonts CDN only
connect-src 'self' https://*.supabase.co https://api.emailjs.com Allows API requests to same origin, Supabase backend, and EmailJS API. All other XHR/fetch/WebSocket destinations are blocked
img-src 'self' data: Allows same-origin images and inline data URIs (used for generated receipt PDFs)
frame-ancestors 'none' Prevents the application from being embedded in any <iframe>, <frame>, or <object> — blocks clickjacking attacks

Actual CSP Meta Tag

<meta http-equiv="Content-Security-Policy"
  content="default-src 'self';
    script-src 'self';
    style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
    font-src https://fonts.gstatic.com;
    connect-src 'self' https://*.supabase.co https://api.emailjs.com;
    img-src 'self' data:;
    frame-ancestors 'none';" />
Why unsafe-inline for styles? React applies component styles via the style attribute on DOM elements. Without 'unsafe-inline' in style-src, all inline styles would be blocked, breaking the UI. This is a known trade-off in React applications that use inline styling patterns rather than CSS-in-JS libraries with nonce support.

Input Sanitization

All user-supplied text is sanitized before being stored or displayed. The sanitization utilities are defined in src/utils/validators.js and applied throughout the application wherever user input is processed.

sanitizeText(value)

The primary text sanitization function, applied to all free-text inputs (student names, remarks, rejection reasons, etc.).

export const sanitizeText = (text) => {
  if (!text) return ''
  let clean = text.replace(/<[^>]*>/g, '').trim()
  // Strip CSV injection chars from start AND dangerous chars throughout
  clean = clean.replace(/^[=+\-@\t\r]+/, '')
  clean = clean.replace(/[\t\r]/g, ' ')
  return clean.substring(0, 500)
}
Step Operation Purpose
1 Strip HTML tags via regex /<[^>]*>/g Prevents stored XSS by removing any HTML elements from the input
2 Trim whitespace Normalizes input by removing leading/trailing whitespace
3 Strip CSV injection prefixes /^[=+\-@\t\r]+/ Prevents CSV injection attacks. Spreadsheet applications (Excel, Google Sheets) interpret cells starting with =, +, -, or @ as formulas, which can execute arbitrary commands when the exported CSV is opened
4 Replace tab/carriage-return characters /[\t\r]/g Prevents CSV field splitting and other control character injection attacks
5 Truncate to 500 characters Caps input length to prevent excessively large payloads and potential DoS via storage exhaustion

sanitizeRollNumber(value)

Specialized sanitizer for student roll number inputs. Applied on blur of the roll number field in the fee submission form.

export const sanitizeRollNumber = (roll) => {
  if (!roll) return ''
  return roll.trim().toUpperCase().replace(/[^A-Z0-9]/g, '')
}
  • Trims whitespace from both ends
  • Converts to uppercase for consistent formatting
  • Strips all characters except A-Z and 0-9 (whitelist approach)

isValidEmail(email)

Regex-based email format validation used on the student email field in the fee submission form and on staff email inputs in the Admin Panel.

export const isValidEmail = (email) => {
  if (!email || email.length > 254) return false
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)
}
  • Rejects empty strings and emails longer than 254 characters (RFC 5321 maximum)
  • Pattern requires: localpart@domain.tld
  • Does not allow spaces or multiple @ signs
  • Used as a client-side gate — the actual email deliverability is validated by EmailJS at send time

PII Sanitization in Activity Logs

The ActivityLogPage.jsx component sanitizes log detail objects before rendering them in the UI. This prevents accidental exposure of Personally Identifiable Information (PII) to staff members viewing the activity log.

const sanitizeDetails = (details) => {
  if (!details || typeof details !== 'object') return details
  const piiKeys = ['email', 'password', 'phone', 'phone_number', 'mobile']
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/
  const cleaned = {}
  for (const [key, value] of Object.entries(details)) {
    if (piiKeys.includes(key.toLowerCase())) continue
    if (typeof value === 'string' && emailRegex.test(value.trim())) continue
    cleaned[key] = value
  }
  return Object.keys(cleaned).length > 0 ? cleaned : null
}
Filter Mechanism Fields Caught
Key-based exclusion Compares key.toLowerCase() against PII key list email, password, phone, phone_number, mobile
Value-based exclusion Tests string values against email regex pattern Any value that looks like an email address, regardless of key name
Client-Side Only: This PII sanitization is performed client-side for display purposes only. The actual log data in the activity_logs database table retains all fields for complete audit trail purposes. The sanitization prevents casual exposure of sensitive data to staff viewing logs in the dashboard, but does not affect the underlying data.

Rate Limiting

Client-side rate limiting is implemented in two critical areas to deter brute-force attacks. Both mechanisms use React component state to track attempt counts and enforce cooldown periods.

Login Lockout (LoginPage.jsx)

Prevents brute-forcing staff login credentials by locking the form after repeated failures.

Parameter Value
Max attempts before lockout 5 failed login attempts
Lockout duration 60 seconds
State variables attempts (integer counter), lockoutUntil (timestamp)
Lockout behavior Submit button disabled, countdown timer displayed in error message
Reset conditions Successful login or page refresh (component remount) 
// On failed login (catch block)
setAttempts(prev => {
  const next = prev + 1
  if (next >= 5) setLockoutUntil(Date.now() + 60_000)
  return next
})

// On form submit (before login attempt)
if (lockoutUntil && Date.now() < lockoutUntil) {
  const secs = Math.ceil((lockoutUntil - Date.now()) / 1000)
  setError(`Too many attempts. Try again in ${secs}s.`)
  return
}

Tracking Rate Limit (TrackingPage.jsx)

Prevents brute-forcing reference numbers by limiting the number of lookups within a time window.

Parameter Value
Max lookups before cooldown 5 tracking lookups
Cooldown duration 30 seconds
State variables trackCount (integer counter), trackCooldown (boolean)
Cooldown behavior Error message displayed, lookup function returns early
Reset conditions Cooldown expires via setTimeout callback (resets both trackCooldown and trackCount) 
setTrackCount(prev => {
  const next = prev + 1
  if (next >= 5) {
    setTrackCooldown(true)
    setTimeout(() => {
      setTrackCooldown(false)
      setTrackCount(0)
    }, 30_000)
  }
  return next
})
Client-Side Limitation: Both rate limiting mechanisms use React component state. They are effective against casual abuse but can be circumvented by refreshing the page (which resets component state) or by making direct API calls that bypass the React UI. For production-grade rate limiting, server-side enforcement via Supabase Edge Functions or an API gateway should be considered.

Role-Based Access Control (RBAC)

The application defines 4 access levels. Access is enforced at three layers: React Router (client-side navigation), ProtectedRoute wrapper (authentication gate), and RLS policies (database-level authorization).

Access Levels

1. Public (No Authentication)

  • LandingPage (/) — Marketing page, no data access
  • StudentFormPage (/submit) — Fee submission form. Uses SECURITY DEFINER RPCs to insert data without authentication
  • TrackingPage (/track) — Reference number lookup. Uses SECURITY DEFINER RPCs to query data without authentication
  • LoginPage (/login) — Staff authentication entry point

2. Class Representative (CR)

  • Dashboard with a single assigned section
  • Approve or reject fee requests within their assigned section
  • View activity logs filtered to their section
  • Export data (CSV, Excel, PDF) for their section

3. Teacher

  • All CR capabilities
  • Access to multiple assigned sections
  • Section switcher component to toggle between assigned sections

4. Admin

  • Full system configuration via the Admin Panel
  • Staff account management: create, edit, deactivate, delete accounts
  • Data management: branches, specialisations, semesters, sections, students, batch configs
  • CSV upload for bulk student data import
  • Activity log viewing across all sections
  • Uses Edge Functions with service_role key for privileged database operations

ProtectedRoute Implementation

The ProtectedRoute component wraps authenticated routes. It checks two conditions:

  1. Authentication: Verifies that user and profile exist in AuthContext. If not, redirects to /login.
  2. Role Authorization: If an allowedRoles prop is provided, checks that profile.role is included in the array. If not, redirects to /.
// Dashboard route — only CR and Teacher
<ProtectedRoute allowedRoles={['cr', 'teacher']}>
  <DashboardPage />
</ProtectedRoute>

// Activity Log route — any authenticated user
<ProtectedRoute>
  <ActivityLogPage />
</ProtectedRoute>
Admin Route Note: The /admin route uses AdminPage directly without the ProtectedRoute wrapper, but AdminPage performs its own internal authorization check. It verifies that the authenticated user has the admin role and redirects unauthorized users.

Optimistic Concurrency Control

When multiple staff members (CRs or Teachers) are assigned to the same section, they may attempt to approve or reject the same fee request simultaneously. The application uses an optimistic concurrency pattern to detect and prevent double-processing.

Mechanism

The approve and reject operations use a conditional update query that includes the current status as a filter:

const { error, count } = await supabase
  .from('fee_requests')
  .update({
    status: 'approved',
    reviewed_at: new Date().toISOString(),
    reviewed_by: profile.id,
  })
  .eq('id', req.id)
  .eq('status', 'pending')       // Only update if still pending
  .select('id', { count: 'exact' })

Race Condition Prevention

Step 1
Staff A clicks "Approve" on request #123
Step 2
UPDATE query filters by id = 123 AND status = 'pending'
Step 3
If count === 1: update succeeded, request is now approved
Step 4
Staff B clicks "Approve" on the same request #123 moments later
Step 5
UPDATE query filters by id = 123 AND status = 'pending', but status is now 'approved'
Step 6
count === 0: no rows matched, indicating the request was already processed

When count === 0, the application displays a warning toast:

if (count === 0) {
  addToast('Request already processed by another reviewer.', 'warning')
  refetch()  // Refresh the request list to reflect the current state
  return
}
Why This Works: PostgreSQL's UPDATE ... WHERE is atomic. When two concurrent UPDATE statements target the same row, one will execute first and change the status from 'pending' to 'approved' (or 'rejected'). The second UPDATE's WHERE clause will then fail to match any rows because the status is no longer 'pending'. The count return value allows the application to detect this scenario without a separate SELECT query.

Edge Function Security (admin-actions)

The admin-actions Edge Function runs on Supabase Edge Functions (Deno runtime) and handles privileged operations that require the service_role key. This key bypasses all RLS policies and has full database access.

Supported Actions

Action Description Input
create_user Creates a new Supabase Auth user and corresponding users table record email, password, full_name, role, staff_id, branch_id, specialisation_id, section assignments
reset_password Resets a staff member's password via Supabase Admin API user_id, new_password
deactivate_user Deactivates a staff account by setting is_active = false in the users table user_id

Security Checks

  1. Authentication: The Edge Function extracts the JWT from the Authorization header and verifies it using Supabase's auth.getUser() method. Unauthenticated requests are rejected with a 401 status.
  2. Authorization: After authentication, the function queries the users table to verify that the caller has the admin role. Non-admin callers are rejected with a 403 status.
  3. Input Validation: The request body is validated for required fields based on the action type. Missing or invalid fields result in a 400 status.
  4. Service Role Isolation: The service_role key is stored as a Supabase Edge Function secret (environment variable). It is never transmitted to or accessible by the client application.
Client Request
Admin Panel sends POST with JWT in Authorization header
Auth Check
Edge Function verifies JWT via auth.getUser()
Role Check
Queries users table to confirm role = 'admin'
Validation
Validates request body fields for the specified action
Execution
Performs the action using service_role Supabase client
Response
Returns success or error JSON to the client
No Client-Side Bypass: The admin-actions Edge Function is the only pathway for creating users, resetting passwords, and deactivating accounts. These operations cannot be performed through the Supabase client library with the anonymous key because RLS policies block direct access to Supabase Auth admin methods. An attacker with access to the anonymous key cannot escalate privileges without also compromising the Edge Function endpoint with a valid admin JWT.

logAction Utility

All log entries are created via the logAction utility in src/utils/logAction.js. This function inserts a row into the activity_logs table using the Supabase client.

export const logAction = async ({
  actionType,
  performedByRole,
  performedByName,
  performedById = null,
  targetName = null,
  targetRoll = null,
  details = null,
  sectionName = null,
  branchName = null,
}) => {
  try {
    const { error } = await supabase.from('activity_logs').insert({
      action_type: actionType,
      performed_by_role: performedByRole,
      performed_by_name: performedByName,
      performed_by_id: performedById,
      target_name: targetName,
      target_roll: targetRoll,
      details: details,
      section_name: sectionName,
      branch_name: branchName,
    })
    if (error) console.error('Failed to log action:', error)
  } catch (err) {
    console.error('Failed to log action:', err)
  }
}

Denormalized Design

The activity_logs table uses a denormalized schema with no foreign key constraints. All contextual data (names, roles, section/branch names) is stored directly as text values rather than as references to other tables. This design decision ensures that:

  • Log entries survive even if the referenced entities (users, students, sections, branches) are subsequently deleted or modified
  • Log queries do not require JOINs, enabling fast read performance
  • The audit trail remains an immutable historical record, unaffected by ongoing data changes

Automatic Cleanup

A pg_cron scheduled job runs periodically to delete log entries older than 31 days. This prevents unbounded table growth while retaining a rolling window of recent activity for audit purposes.

Fail-Safe Logging: The logAction utility wraps the insert in a try/catch block and only logs errors to the console. A logging failure never prevents the primary operation from completing. This ensures that a temporary database connectivity issue with the logging subsystem does not block user actions such as approving or rejecting fee requests.

Password Security

Staff passwords are managed through Supabase Auth, which handles hashing, storage, and verification. The application adds additional security through its password generation algorithm.

Password Hashing

Supabase Auth uses bcrypt for password hashing. Passwords are hashed before storage and never stored in plaintext. The application never has access to the raw password after the initial creation or reset operation.

Auto-Generated Passwords

When creating a new staff account or resetting a password, the Admin Panel generates a cryptographically secure random password using the generatePassword() function in StaffManager.jsx.

const secureRandom = (max) => {
  const arr = new Uint32Array(1)
  crypto.getRandomValues(arr)
  return arr[0] % max
}

const generatePassword = () => {
  const upper = 'ABCDEFGHJKLMNPQRSTUVWXYZ'
  const lower = 'abcdefghjkmnpqrstuvwxyz'
  const digits = '23456789'
  const symbols = '!@#$%&*'
  const all = upper + lower + digits + symbols
  const pick = (s) => s[secureRandom(s.length)]
  let pw = [pick(upper), pick(lower), pick(digits), pick(symbols)]
  for (let i = 0; i < 8; i++) pw.push(pick(all))
  // Fisher-Yates shuffle
  for (let i = pw.length - 1; i > 0; i--) {
    const j = secureRandom(i + 1);
    [pw[i], pw[j]] = [pw[j], pw[i]]
  }
  return pw.join('')
}

Password Generation Properties

Property Detail
Length 12 characters (4 guaranteed category picks + 8 random from all categories)
Uppercase requirement At least 1 character from ABCDEFGHJKLMNPQRSTUVWXYZ
Lowercase requirement At least 1 character from abcdefghjkmnpqrstuvwxyz
Digit requirement At least 1 character from 23456789
Special character requirement At least 1 character from !@#$%&*
Ambiguous characters excluded I, l, O, 0, 1 are omitted from character pools to avoid visual confusion
Randomness source crypto.getRandomValues() (Web Crypto API, cryptographically secure)
Position randomness Fisher-Yates shuffle ensures that the guaranteed category characters are not always at the start of the password

No Self-Service Password Reset: Staff members cannot reset their own passwords. Only admin users can reset passwords via the Admin Panel, which invokes the admin-actions Edge Function with the reset_password action. This centralizes password management and prevents unauthorized password changes.

Build-Time Security

The Vite build configuration (vite.config.js) includes production-specific optimizations that remove debug information from the production bundle, reducing the attack surface and preventing information leakage.

export default defineConfig(({ mode }) => ({
  plugins: [react(), tailwindcss()],
  esbuild: mode === 'production'
    ? {
        drop: ['debugger'],
        pure: [
          'console.log',
          'console.debug',
          'console.info',
          'console.warn',
        ],
      }
    : {},
  build: {
    sourcemap: false,
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['react', 'react-dom', 'react-router-dom'],
          supabase: ['@supabase/supabase-js'],
        },
      },
    },
  },
}))

Production Stripping

Setting Effect Security Benefit
esbuild.pure Removes console.log, console.debug, console.info, console.warn calls from the production bundle Prevents leaking internal application state, variable values, API responses, and debug messages to the browser console
esbuild.drop: ['debugger'] Removes all debugger statements from the production bundle Prevents accidental breakpoints in production that could halt execution and expose the call stack to anyone with DevTools open
build.sourcemap: false Disables source map generation for production builds Prevents reverse-engineering of the original source code structure, variable names, and component logic from the minified production bundle
Development vs Production: The esbuild stripping configuration is conditionally applied only when mode === 'production'. During local development (pnpm run dev), all console statements and debugger calls remain functional to support debugging and development workflows.
console.error Preserved: Note that console.error is intentionally not included in the pure list. Error logging remains active in production to facilitate debugging of critical failures in the browser console. The logAction utility, for instance, falls back to console.error when an activity log insert fails.
