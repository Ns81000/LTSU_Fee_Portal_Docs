Dashboard Overview

The Staff Dashboard is the primary operational interface for Class Representatives (CRs) and Teachers. It is served at the /dashboard route and is protected by the ProtectedRoute component with allowedRoles: ['cr', 'teacher']. Any unauthenticated user or user with a role outside this list is redirected to the login page.

DashboardPage.jsx is approximately 500 lines of React code and constitutes the single largest page component in the application. It orchestrates statistics computation, tabbed fee request views, search and sort filtering, pagination, approve/reject workflows, batch operations, multi-format data export, and payment status tracking -- all within a responsive layout that adapts between desktop and mobile viewports.

Layout Structure

The dashboard uses a dual-layout approach that fundamentally changes the navigation paradigm based on viewport width.

Desktop Layout (above 768px)

  • Sidebar (fixed left): A permanently visible sidebar rendered by Sidebar.jsx. Contains three navigation links: Dashboard (active), Activity Log, and Logout. At the top, the sidebar displays the staff member's full_name, a role badge (either CR or Teacher), and their assigned section information (branch name, section name, semester number).
  • Navbar (fixed top): Rendered by Navbar.jsx, displays the "LTSU Fee Portal" title across the top of the main content area. On desktop, this sits to the right of the sidebar.
  • Main content area: Fills the remaining viewport space to the right of the sidebar and below the navbar. Contains all dashboard content: stats cards, section switcher, tabs, tables, and export controls.

Mobile Layout (768px and below)

  • Navbar (fixed top): Same Navbar.jsx component, now spanning full width.
  • Main content area: Full-width, scrollable content below the navbar and above the mobile nav.
  • MobileNav (fixed bottom): Rendered by MobileNav.jsx, a bottom tab bar with three tabs: Dashboard (grid icon), Log (list icon), and Logout (log-out icon). Each tab includes an SVG icon and a label. The active tab is highlighted with the gold accent colour.

Section Switcher (SectionSwitcher.jsx)

The Section Switcher is a dedicated component that appears only when the logged-in user is a Teacher who has been assigned to more than one section. CRs are always bound to a single section and will never see this component.

Rendering Logic

  • The component receives the teacher's assignedSections array and the current activeSectionId as props.
  • If the array has fewer than 2 entries, the component returns null (renders nothing).
  • For each section, a pill button is rendered showing: section name + branch name + semester number.

Visual Behaviour

  • The active section pill receives a gold background (#c8a951) with dark text.
  • Inactive pills have a transparent background with a subtle border and lighter text.
  • Clicking an inactive pill fires the onSectionChange callback, which updates activeSectionId in the parent DashboardPage state.
  • When the active section changes, useFeeRequests re-fetches data for the newly selected section, causing the statistics, tables, and all derived views to update.

Statistics Cards

Four StatsCard components are rendered in a horizontal row across the top of the dashboard content area. Each card displays a label, a computed numeric value, and a colour-coded icon. These statistics are recalculated on every data change (section switch, approve/reject, refetch).

#CardComputationDetails
1 Total Students students.length Count of all students in the currently active section. This includes both active and inactive students returned by the useFeeRequests hook.
2 Submitted feeRequests.length Count of all fee requests (any status: pending, approved, or rejected) associated with students in the current section. Represents the total number of submissions received.
3 Pending feeRequests.filter(r => r.status === 'pending').length Count of fee requests that are still awaiting review. This is the primary metric staff use to determine their current workload.
4 Total Collected sum of fee_amount for approved requests Iterates over all fee requests with status === 'approved', sums their fee_amount values, and formats the result as INR currency using the formatCurrency utility (e.g., "12,500.00").
Currency Formatting: The formatCurrency utility from utils/formatters.js uses Intl.NumberFormat('en-IN', { style: 'currency', currency: 'INR' }) to produce locale-aware Indian Rupee formatting with commas at the thousands and lakhs positions.

Search

A text input field above the table allows staff to filter results by student name or roll number. The search value is debounced at 300ms using the useDebounce hook from hooks/useDebounce.js. The debounced value is applied as a case-insensitive substring match against both full_name and roll_number fields. An empty search string shows all records for the active tab.

Sort

A dropdown select allows sorting by:

  • Name (A-Z / Z-A): Alphabetical sort on full_name
  • Roll Number (Asc / Desc): Alphanumeric sort on roll_number
  • Date (Newest / Oldest): Chronological sort on created_at or reviewed_at depending on the tab
  • Amount (High / Low): Numeric sort on fee_amount

Payment Filter (Approved Tab Only)

When the Approved tab is active, an additional dropdown filter appears with three options:

  • All: Shows all approved requests
  • Paid: Shows only requests where the student ID is in the paidStudentIds Set
  • Unpaid: Shows only requests where the student ID is not in the paidStudentIds Set

Pagination

Results are paginated at 25 records per page. The pagination control renders the following buttons:

  • First: Jumps to page 1. Disabled when already on page 1.
  • Prev: Goes to the previous page. Disabled on page 1.
  • Numbered pages: Direct page number buttons. The current page is highlighted. A window of pages is shown around the current page to prevent rendering hundreds of buttons for large datasets.
  • Next: Goes to the next page. Disabled on the last page.
  • Last: Jumps to the final page. Disabled when already on the last page.

Changing tabs, search terms, or sort order resets the page back to 1.

Approve / Reject Workflow

The approve and reject operations are the most critical workflows in the dashboard. Both implement optimistic concurrency control to prevent race conditions when multiple staff members review the same request simultaneously.

Single Approve

When a staff member clicks the "Approve" button on a pending fee request, the following sequence executes:

1. Staff clicks "Approve" on a pending request row
2. Supabase update with optimistic concurrency lock: supabase.from('fee_requests').update({ status: 'approved', reviewed_by: user.id, reviewed_at: new Date().toISOString() }).eq('id', requestId).eq('status', 'pending').select('id', { count: 'exact' })
3. Check count: if count === 0, the request was already processed by another user -- show warning toast and abort
4. On success: send approval email via EmailJS (template ID: template_bsick8e)
5. Update receipt_sent flag to true on the fee_request record
6. Log activity with action_type: 'approve_request'
7. Call refetch() to refresh all data from Supabase

Optimistic Concurrency Control

The key technique here is the double .eq() clause: .eq('id', requestId).eq('status', 'pending'). This ensures the update only succeeds if the request is still in 'pending' status at the moment the database processes the query. If another staff member has already approved or rejected the same request, the .eq('status', 'pending') condition fails, no rows are updated, and the returned count is 0. The dashboard then shows a warning toast informing the staff member that the request was already processed, preventing duplicate approvals or conflicting status changes.

Single Reject

When a staff member clicks the "Reject" button, a modal dialog opens to collect the rejection reason before proceeding.

1. Staff clicks "Reject" on a pending request row
2. RejectModal opens: textarea for rejection_reason input
3. Validation: minimum 10 characters, maximum 500 characters. The confirm button is disabled until the input passes validation.
4. On confirm: Supabase update with optimistic concurrency lock: supabase.from('fee_requests').update({ status: 'rejected', reviewed_by: user.id, reviewed_at: new Date().toISOString(), rejection_reason: reason }).eq('id', requestId).eq('status', 'pending').select('id', { count: 'exact' })
5. Check count: if count === 0, the request was already processed -- show warning toast and close modal
6. On success: send rejection email via EmailJS (template ID: template_wd7lk49)
7. Log activity with action_type: 'reject_request'
8. Call refetch() to refresh all data from Supabase

RejectModal Component

The RejectModal component (components/forms/RejectModal.jsx) renders a modal overlay with:

  • A header displaying "Reject Fee Request"
  • A <textarea> for the rejection reason with a placeholder guiding the staff to explain why the request is being rejected
  • A character counter showing current length vs. the 500-character maximum
  • A "Cancel" button that closes the modal without any side effects
  • A "Confirm Reject" button that is disabled until the reason text is between 10 and 500 characters (inclusive)

Concurrency Edge Case: If two staff members open the same pending request at the same time and both click "Approve", only the first update will succeed. The second staff member's update will return count === 0, and they will see a warning toast: "This request has already been processed by another user." Batch Approve

The batch approve feature allows staff to select multiple pending fee requests and approve them all in a single operation. This is designed for high-volume scenarios where a staff member needs to process many requests efficiently.

Selection Mechanism

  • Each row in the Pending tab has a checkbox in the first column.
  • The table header row includes a "select all" checkbox that toggles all currently visible (paginated) rows.
  • Selected request IDs are stored in a local state array.
  • An "Approve Selected" button appears above the table when at least one request is selected.
  • The button displays the count of selected requests (e.g., "Approve Selected (12)").

Batch Limit

A maximum of 25 requests can be batch-approved at once. If the staff member selects more than 25 requests, a warning toast is shown and the operation is blocked. This limit exists to prevent timeouts on the email dispatch loop and to keep the RPC call payload within reasonable bounds.

Execution Flow

1. Staff selects multiple pending requests via checkboxes (max 25)
2. Staff clicks "Approve Selected" button
3. Dashboard calls RPC: supabase.rpc('batch_approve_requests', { p_request_ids: selectedIds, p_reviewer_id: user.id })
4. The RPC function atomically updates all selected requests to status 'approved', sets reviewed_by and reviewed_at
5. For each successfully approved request, an individual approval email is sent via EmailJS (template_bsick8e)
6. Activity is logged as a batch operation
7. Success toast: "Successfully approved N requests"
8. Selection state is cleared and data is refetched

RPC Function: batch_approve_requests

ParameterTypeDescription
p_request_idsuuid[]Array of fee_request IDs to approve
p_reviewer_iduuidThe ID of the staff member performing the batch approve

The server-side RPC function ensures atomicity: either all specified requests are updated, or the entire operation is rolled back. This prevents partial batch approvals that could leave the system in an inconsistent state. The function also only updates requests that are currently in 'pending' status, applying the same optimistic concurrency guard used by single approvals.

Email Dispatch: After the batch RPC succeeds, the dashboard iterates over each approved request and sends an individual approval email. This means a batch of 25 approvals results in 25 separate EmailJS API calls. The emails are sent sequentially (not in parallel) to respect EmailJS rate limits and to ensure each email's success/failure is tracked independently.

Payment Status Tracking

The dashboard provides a client-side mechanism for staff to track which approved students have physically paid their fees (e.g., submitted cash or completed a bank transfer that has been verified outside the system).

Implementation

  • State: paidStudentIds is a Set stored in React component state. It contains the student IDs of students whose payment has been marked as received.
  • Toggle: On the Approved tab, each row displays a toggle button. Clicking the button adds the student ID to the Set (marking as "Paid") or removes it (marking as "Unpaid").
  • Visual indicator: A green Paid badge is shown for students in the Set. A gray Unpaid badge is shown for students not in the Set.

Persistence

Not Persisted: The payment status is tracked entirely in client-side React state. It is not written to the Supabase database. This means the Paid/Unpaid status resets when the page is refreshed, when the staff member logs out, or when they switch sections. This is a deliberate design decision -- the LTSU Fee Portal tracks fee request approvals, not actual payment collection. Physical payment verification is handled outside the system.

Usage in Exports

When exporting data via PDF, Excel, or CSV, the current payment status state is included in the exported data. This means if a staff member marks several students as "Paid" and then exports the Approved tab, the export will reflect those Paid/Unpaid values. However, since the state is ephemeral, re-exporting after a page refresh will show all students as "Unpaid" again.

Excel Export (exportExcel.js)

  • Library: SheetJS (xlsx)
  • Workbook structure: Two sheets are created within a single .xlsx file:
SheetContents

CSV Export (exportCSV.js)

  • Encoding: UTF-8 with BOM (Byte Order Mark: \uFEFF) prepended to the output. The BOM ensures Microsoft Excel correctly interprets the file as UTF-8 rather than defaulting to ANSI encoding, which would mangle special characters.
  • Cell sanitization: Before writing any cell value, the export function checks if the value starts with any of the characters =, +, -, or @. If so, a single quote (') is prepended to the cell value. This prevents CSV injection attacks where a malicious cell value could execute formulas when opened in a spreadsheet application.
  • Delimiter: Standard comma (,) delimiter with values enclosed in double quotes.
  • File name: fee-report-{sectionName}-{date}.csv
CSV Injection Prevention: Without cell sanitization, a student name like =CMD("calc") could execute arbitrary commands when the CSV is opened in Excel. The single-quote prefix ('=CMD("calc")) tells Excel to treat the cell as a text literal rather than a formula. This protection applies to all cell values across all columns, not just name fields.

Mobile Responsiveness

Below the 768px breakpoint, the dashboard undergoes significant layout and component changes to provide a usable experience on smaller screens.

Navigation Changes

  • The desktop Sidebar component is hidden entirely (display: none).
  • The MobileNav component appears as a fixed bottom bar with three tabs: Dashboard (grid icon), Log (list icon), and Logout (log-out icon).
  • The active tab is indicated by the gold accent colour on both the icon and the label text.

Table to Card Transformation

Data tables are replaced with card-based layouts optimised for vertical scrolling on mobile devices. Two specialised card components handle this transformation:

MobileRequestCard

  • Used for Pending, Approved, and Rejected tabs.
  • Each card displays: student name (bold header), roll number, status badge, fee amount, and tab-specific details (submission date, reviewer, rejection reason, etc.).
  • Action buttons (Approve/Reject) are rendered at the bottom of the card as full-width buttons for easy tap targets.
  • Wrapped in React.memo to prevent unnecessary re-renders when other cards in the list update.

MobileStudentCard

  • Used for the Students tab.
  • Each card displays: student name (bold header), roll number, batch year, and fee status badge.
  • Simpler layout than MobileRequestCard since there are no action buttons.
  • Also wrapped in React.memo for performance optimisation.
React.memo Optimisation: Both mobile card components use React.memo with the default shallow comparison. Since fee request objects are replaced (not mutated) on each refetch, this effectively prevents re-rendering cards whose data has not changed. In a section with 200 students, approving one request only causes a re-render of the affected card and the stats display, not all 200 cards.

Additional Mobile Adjustments

  • Stats cards stack vertically (single column) instead of the four-column horizontal layout.
  • The search input and sort dropdown stack vertically.
  • Pagination buttons use smaller sizing to fit within the viewport.
  • The Section Switcher pills wrap to multiple lines if there are many sections.
  • The Export Menu dropdown aligns to the left edge to prevent overflow.

useFeeRequests Hook

The useFeeRequests custom hook (hooks/useFeeRequests.js) is the primary data-fetching mechanism for the dashboard. It encapsulates all Supabase queries needed to populate the dashboard with student and fee request data.

Parameters

ParameterTypeDescription
sectionIdsstring[]Array of section UUIDs to fetch data for. For CRs, this is a single-element array. For teachers, it contains the currently active section ID.
batchConfigsobject[]Array of batch configuration objects from DataContext. Each object contains batch_year and fee_amount fields used to enrich student records.

Data Fetching Process

1. Fetch students: supabase.from('students').select('*').in('section_id', sectionIds)
2. Enrich students: for each student, find the matching batchConfig by batch_year and attach the fee_amount to the student object
3. Fetch fee requests: supabase.from('fee_requests').select('*, students(*), semesters(name, number), reviewer:reviewed_by(full_name)').in('student_id', studentIds)
4. Return: { students (enriched), feeRequests, loading, error, refetch }

Return Value

FieldTypeDescription
studentsobject[]Array of student objects enriched with fee_amount from the matching batch config. If no batch config matches a student's batch_year, fee_amount defaults to 0.
feeRequestsobject[]Array of fee request objects with joined data: nested students object, nested semesters object (name, number), and nested reviewer object (full_name of the staff who reviewed it, or null if pending).
loadingbooleantrue while either the students query or the fee requests query is in progress. The dashboard renders a Spinner overlay while loading.
errorstring | nullError message if either query fails. Displayed as an error toast in the dashboard.
refetchfunctionCallback to re-execute both queries. Called after every approve, reject, or batch approve operation to refresh the displayed data.

Fee Request Join Structure

The fee requests query uses Supabase's nested select syntax to join related tables in a single query:

supabase
  .from('fee_requests')
  .select(`
    *,
    students (*),
    semesters (name, number),
    reviewer:reviewed_by (full_name)
  `)
  .in('student_id', studentIds)

AbortController Cleanup

The hook creates an AbortController instance at the start of each fetch cycle. The controller's signal is not passed directly to Supabase (which does not natively support AbortController), but is used to prevent state updates after the component unmounts or when a new fetch cycle begins. When the effect cleanup runs (on unmount or before a re-fetch), the controller is aborted and a flag is set. If the Supabase query completes after the abort, the state-setting calls are skipped, preventing the "Can't perform a React state update on an unmounted component" warning.

Re-fetch Triggers: The hook's useEffect dependency array includes sectionIds and batchConfigs. Changing the active section (via SectionSwitcher) causes sectionIds to change, which triggers a re-fetch. The manual refetch() function is used after approve/reject operations. Both paths go through the same AbortController cleanup logic.
