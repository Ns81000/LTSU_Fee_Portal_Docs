Schema Overview

The LTSU Fee Portal database is powered by Supabase PostgreSQL and consists of 10 relational tables within the public schema. Every table has Row-Level Security (RLS) enabled, ensuring that data access is controlled at the database layer rather than relying solely on application logic. Foreign key constraints enforce referential integrity across the schema, and server-side RPC functions encapsulate critical business logic such as fee request submission, duplicate checking, and batch approval.

The schema is organized around three core domains:

  • Academic structure: branches, specialisations, sections, semesters, and batch_configs define the institutional hierarchy. Branches contain specialisations, sections are scoped to a branch + specialisation + semester combination, and batch configurations set per-year fee amounts.
  • User management: users and user_sections manage staff accounts (admin, teacher, CR roles) and their section assignments. The users table links to Supabase Auth via auth.users for authentication.
  • Fee processing: students and fee_requests handle the core business process. Students are linked to their academic placement, and fee requests track the full lifecycle from submission through approval or rejection. activity_logs captures an audit trail of all administrative actions.

Table: users

The users table stores all staff accounts that can log into the admin or dashboard areas of the LTSU Fee Portal. Each row corresponds to a Supabase Auth user, linked by the id column which references auth.users.id. This table does not store passwords -- authentication is handled entirely by Supabase Auth.

The role column determines the user's access level within the application. There are three possible roles: admin (full access to all features, including the admin panel and Edge Function invocations), teacher (dashboard access scoped to their assigned sections), and cr (class representative -- dashboard access scoped to their assigned section with limited actions).

The created_by column is a self-referencing foreign key that records which admin user created this account. This enables a full audit trail of user provisioning.

Column Type Nullable Default Constraints Description
id uuid NO -- PRIMARY KEY References auth.users.id Supabase Auth user ID. Not auto-generated -- set to match the auth.users row.
role text NO -- CHECK role IN ('admin', 'cr', 'teacher') User role determining access level in the application.
full_name text NO -- -- Display name of the staff member.
staff_id text NO -- UNIQUE Institutional staff identifier (e.g., employee code).
email text NO -- UNIQUE Email address, must match the corresponding auth.users email.
is_active boolean NO true -- Soft-delete flag. Inactive users cannot log in.
created_at timestamptz NO now() -- Timestamp of user creation.
created_by uuid YES NULL FK users.id Self-referencing FK to the admin who created this account.

Table: students

The students table contains all student records uploaded via the CSV Uploader in the Admin Panel. Students are not authentication users -- they do not have Supabase Auth accounts or login credentials. Instead, their identity is verified by roll number lookup during the fee request submission process on the public Student Form page.

Each student belongs to a branch, specialisation, and section, which are resolved via foreign key references. The batch_year integer determines which batch_configs fee amount applies to the student. The uploaded_by column tracks which admin uploaded the student (typically via CSV bulk import).

Primary keys use gen_random_uuid() for automatic UUID generation on insert.

Table: fee_requests

The fee_requests table is the central table of the LTSU Fee Portal. It tracks the complete lifecycle of every fee request from initial submission through approval or rejection. Each row represents a single fee payment request submitted by a student for a specific semester.

The reference_number column is auto-populated by a database trigger on insert, generating a unique identifier in the format LTSU-YYYY-XXXXX where YYYY is the current year and XXXXX is a zero-padded sequential number. Students use this reference number to track the status of their request on the public Tracking page.

The status column tracks the request lifecycle with three possible states: pending (awaiting review), approved (fee confirmed by a teacher or CR), and rejected (fee declined with a mandatory rejection reason). The reviewed_at, reviewed_by, and rejection_reason columns are populated when the request moves out of the pending state.

The receipt_sent flag tracks whether an email receipt has been sent after approval, preventing duplicate email delivery.

Table: activity_logs

The activity_logs table serves as a comprehensive audit trail for all administrative actions performed within the LTSU Fee Portal. Unlike most other tables in the schema, activity_logs is intentionally denormalized and does not use foreign key constraints. This design decision ensures that log entries remain intact even if the referenced users, students, sections, or branches are later deleted or modified.

Each log entry captures the action type (e.g., approve_request, reject_request, upload_students, create_user), who performed it, who or what was affected, and additional contextual details stored in a flexible jsonb column. The ip_address column records the client IP when available.

Entries older than 31 days are automatically purged by a pg_cron scheduled job to prevent unbounded table growth.

Denormalized by Design: The activity_logs table stores names, roles, and section/branch names as plain text rather than foreign key references. This ensures log entries remain readable and intact even after referenced entities are deleted, deactivated, or renamed. The trade-off is potential data duplication, which is acceptable for an audit log that is automatically pruned after 31 days.

Table: branches

The branches table defines the academic departments or branches offered by the institution (e.g., Computer Science, Mechanical Engineering). Each branch has a unique human-readable name and a short code (e.g., "CS", "ME") used for display and CSV matching purposes.

Branches are the top-level organizational unit in the academic hierarchy. Specialisations belong to branches, and sections are scoped to a branch + specialisation + semester combination. Students are assigned to a branch upon CSV upload.

Table: specialisations

The specialisations table defines sub-divisions within a branch. For example, the "Computer Science and Engineering" branch might have specialisations like "Artificial Intelligence & Machine Learning" or "Data Science". Each specialisation belongs to exactly one branch via the branch_id foreign key.

Specialisations have both a name (full display name) and a code (short identifier used in CSV imports and compact displays). Unlike branches, the name and code are not globally unique -- they are expected to be unique within the context of their parent branch, though this is enforced at the application level rather than via a database constraint.

Table: sections

The sections table represents class sections, which are the primary organizational unit for assigning teachers/CRs and grouping students for fee management. Each section is uniquely identified by the combination of its branch_id, specialisation_id, and semester_id along with a name (e.g., "A", "B", "C").

Sections are the key linkage point between users and students. Teachers and CRs are assigned to sections via the user_sections junction table, and students belong to exactly one section. The dashboard view is scoped by section, showing only fee requests for students in the user's assigned sections.

Table: semesters

The semesters table is a simple lookup table defining the available semesters in the academic structure. Each semester has a human-readable name (e.g., "Semester 1") and a numeric number (e.g., 1) used for sorting and programmatic access. Both name and number are unique.

Unlike most other configuration tables, semesters do not have a created_by column because they are typically seeded once during initial database setup and rarely modified afterward.

Table: batch_configs

The batch_configs table stores per-year fee configurations. Each row maps a batch_year (enrollment year) to a fee_amount in INR. When a student submits a fee request, the system looks up the fee amount from this table based on the student's batch_year.

The batch_year column has a UNIQUE constraint, ensuring only one fee configuration exists per enrollment year. Administrators create and manage these configurations through the Batch Manager component in the Admin Panel.

Table: user_sections

The user_sections table is a junction table (many-to-many) that maps staff users to their assigned sections. A single teacher or CR can be assigned to multiple sections, and a single section can have multiple staff members assigned to it. This relationship determines which fee requests appear on each user's dashboard.

When a teacher or CR logs in and views the dashboard, the application queries user_sections to determine their assigned sections, then fetches fee requests for students in those sections. The assigned_by column provides an audit trail of which admin made the assignment.

Foreign Key Relationships

The schema contains 19 foreign key constraints that enforce referential integrity across all tables. Every foreign key reference points to the id column of the target table, which is always a UUID primary key. Foreign keys ensure that orphaned records cannot exist -- for example, a student cannot reference a branch that does not exist, and a fee request cannot reference a student that has been hard-deleted.

The users table has a special self-referencing foreign key on created_by, allowing the schema to track which administrator created each user account. The user_sections table has two distinct foreign keys pointing to users.id -- one for the assigned user (user_id) and one for the assigning admin (assigned_by).

Note that activity_logs intentionally has zero foreign key constraints to ensure audit records persist regardless of changes to referenced entities.

Row-Level Security

Row-Level Security (RLS) is enabled on all 10 tables in the LTSU Fee Portal database. RLS is a PostgreSQL feature that restricts which rows a given user can read, insert, update, or delete based on security policies evaluated at query time. Supabase enforces RLS on every request that uses the anon or authenticated API keys, meaning application clients cannot bypass these restrictions.

The LTSU Fee Portal uses a layered approach to RLS:

  • Public (anonymous) access: The public Student Form and Tracking pages do not use authenticated sessions. Instead, they call SECURITY DEFINER RPC functions (e.g., submit_fee_request, track_fee_request, check_duplicate_fee_request) that execute with elevated privileges, bypassing RLS. This allows anonymous users to submit and track fee requests without needing database-level read/write policies for the anon role.
  • Authenticated staff access: Teachers and CRs access the dashboard using their authenticated session. RLS policies on fee_requests, students, and related tables use auth.uid() to verify that the requesting user is an active staff member. Section-scoped queries are filtered by joining against user_sections at the application level, though RLS ensures that only authenticated users can query at all.
  • Admin operations: Administrative actions that require elevated privileges (such as creating users in Supabase Auth and the users table simultaneously) are performed via a Supabase Edge Function that uses the service_role key. The service_role key bypasses RLS entirely, allowing the Edge Function to perform operations that no client-side policy would permit.
RLS Bypass Mechanisms: There are exactly three scenarios where RLS is bypassed: (1) SECURITY DEFINER RPC functions for public form submission and tracking, (2) the service_role key used by the Edge Function for admin operations, and (3) pg_cron jobs that run as the database owner. In all other cases, RLS policies are strictly enforced.

Key RPC Functions

The LTSU Fee Portal defines four critical server-side RPC (Remote Procedure Call) functions in PostgreSQL. These functions encapsulate complex business logic that must execute atomically and securely on the database server. Three of the four functions use the SECURITY DEFINER modifier, meaning they execute with the privileges of the function owner (the database owner) rather than the calling user. This allows anonymous users to submit and track fee requests without needing direct table-level access.

1. check_duplicate_fee_request

Checks whether a student already has a fee request for a given semester. This is called before submission to prevent duplicate requests and provide user-friendly feedback.

Signature

CREATE FUNCTION check_duplicate_fee_request(
  p_student_id uuid,
  p_semester_id uuid
) RETURNS text
LANGUAGE plpgsql
SECURITY DEFINER

Return Values

  • 'pending' -- A fee request for this student/semester combination exists and is still pending review.
  • 'approved' -- A fee request for this student/semester combination has already been approved.
  • 'rejected' -- A fee request for this student/semester combination was previously rejected. The student may resubmit.
  • 'none' -- No existing fee request found. The student may proceed with submission.

Behavior

Queries the fee_requests table for any row matching the given student_id and semester_id. If a match is found, returns the status value of that request. If no match exists, returns 'none'. The function uses SECURITY DEFINER to allow anonymous callers (the public student form) to check for duplicates without direct SELECT access on fee_requests.

2. submit_fee_request

Handles the complete fee request submission workflow. This is the primary entry point for students submitting fee payments through the public form.

Signature

CREATE FUNCTION submit_fee_request(
  p_student_id uuid,
  p_roll_number text,
  p_student_email text,
  p_semester_id uuid,
  p_batch_year integer,
  p_fee_amount numeric
) RETURNS text
LANGUAGE plpgsql
SECURITY DEFINER

Return Value

Returns the auto-generated reference_number as a text string in the format LTSU-YYYY-XXXXX (e.g., LTSU-2026-00042). This reference number is displayed to the student and used for tracking.

Behavior

Inserts a new row into the fee_requests table with the provided parameters. The reference_number is auto-generated by a database trigger that fires on INSERT (see Database Triggers). After the insert, the function reads back the generated reference_number and returns it to the caller. Uses SECURITY DEFINER to allow anonymous callers to insert into fee_requests without direct INSERT access.

3. track_fee_request

Retrieves the current status and details of a fee request by its reference number. Used by the public Tracking page to allow students to check their request status.

Signature

CREATE FUNCTION track_fee_request(
  p_reference_number text
) RETURNS TABLE(
  reference_number text,
  student_name text,
  roll_number text,
  status text,
  fee_amount numeric,
  submitted_at timestamptz,
  reviewed_at timestamptz,
  rejection_reason text
)
LANGUAGE plpgsql
SECURITY DEFINER

Behavior

Joins fee_requests with students to return the request details along with the student's full name. Returns exactly one row if the reference number matches, or an empty result set if not found. The student name is resolved from the students table via the student_id foreign key. Uses SECURITY DEFINER to allow anonymous callers to read request status without direct SELECT access on fee_requests or students.

4. batch_approve_requests

Approves multiple fee requests in a single atomic operation. Used by the dashboard's bulk approval feature where teachers or CRs can select multiple pending requests and approve them all at once.

Signature

CREATE FUNCTION batch_approve_requests(
  p_request_ids uuid[],
  p_reviewer_id uuid
) RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER

Behavior

Updates all fee_requests rows whose id is in the p_request_ids array: sets status to 'approved', reviewed_at to now(), and reviewed_by to p_reviewer_id. The operation is atomic -- either all requests are approved or none are (if an error occurs). Uses SECURITY DEFINER to ensure the update succeeds regardless of the caller's RLS permissions, since the function verifies authorization via the p_reviewer_id parameter.

Database Triggers

The database uses a trigger to automatically generate human-readable reference numbers for fee requests upon insertion. This ensures that every fee request receives a unique, predictable identifier that students can use for tracking purposes.

Reference Number Generation Trigger

A BEFORE INSERT trigger on the fee_requests table fires a trigger function that generates a reference number in the format LTSU-YYYY-XXXXX:

  • LTSU -- Fixed prefix identifying the institution.
  • YYYY -- The four-digit current year at the time of insertion (e.g., 2026).
  • XXXXX -- A zero-padded five-digit sequential number that resets each year. The trigger queries the maximum existing sequence number for the current year and increments by one.

Example reference numbers: LTSU-2026-00001, LTSU-2026-00002, LTSU-2026-00137.

Trigger Definition (Conceptual)

CREATE OR REPLACE FUNCTION generate_reference_number()
RETURNS trigger AS $$
DECLARE
  current_year text := extract(year FROM now())::text;
  next_seq integer;
BEGIN
  SELECT COALESCE(MAX(
    CAST(SPLIT_PART(reference_number, '-', 3) AS integer)
  ), 0) + 1
  INTO next_seq
  FROM fee_requests
  WHERE reference_number LIKE 'LTSU-' || current_year || '-%';

  NEW.reference_number := 'LTSU-' || current_year || '-' || LPAD(next_seq::text, 5, '0');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER set_reference_number
  BEFORE INSERT ON fee_requests
  FOR EACH ROW
  EXECUTE FUNCTION generate_reference_number();
Sequence Behavior: The reference number sequence resets to 00001 at the start of each calendar year. Within a year, the sequence is monotonically increasing and gap-free under normal operation. If a row is inserted and then deleted, the sequence number is not reused -- the next insert will receive the next higher number.

pg_cron Automated Jobs

The LTSU Fee Portal uses the pg_cron PostgreSQL extension to schedule automated maintenance tasks. pg_cron is a cron-based job scheduler that runs directly inside PostgreSQL, executing SQL commands on a configurable schedule without requiring an external scheduler or application server.

Job: cleanup-old-activity-logs

Property Value
Job Name cleanup-old-activity-logs
Schedule Daily at 3:00 AM UTC (0 3 * * *)
SQL Command DELETE FROM activity_logs WHERE created_at < now() - interval '31 days'
Purpose Prevents unbounded growth of the activity_logs table by purging entries older than 31 days.
RLS Impact pg_cron jobs run as the database owner, bypassing RLS. No special policy is needed.

Cron Schedule Registration

SELECT cron.schedule(
  'cleanup-old-activity-logs',
  '0 3 * * *',
  $$ DELETE FROM activity_logs WHERE created_at < now() - interval '31 days' $$
);
Data Retention: Activity log entries are permanently deleted after 31 days. There is no archive or backup mechanism for old log entries. If long-term audit retention is required, consider exporting logs before the 31-day window via the Activity Log page's export functionality (CSV, Excel, or PDF).

Indexes

PostgreSQL automatically creates indexes for primary key constraints and unique constraints, so the LTSU Fee Portal schema benefits from automatic indexing on all key lookup columns without explicit index definitions. The following indexes are implicitly created by the schema's constraints:

Automatic Indexes (Primary Keys)

A unique B-tree index is automatically created on the primary key column (id) of every table. These indexes support:

  • Fast single-row lookups by UUID (e.g., fetching a student by ID)
  • Efficient foreign key validation when inserting referencing rows
  • JOIN acceleration when resolving foreign key relationships

Foreign Key Columns

PostgreSQL does not automatically create indexes on foreign key columns (unlike primary and unique constraints). However, the query planner benefits significantly from indexes on FK columns used in JOINs. For the LTSU Fee Portal's scale (typically hundreds to low thousands of rows), the lack of explicit FK indexes is unlikely to cause performance issues. If the dataset grows significantly, consider adding indexes on high-traffic FK columns such as:

  • students.branch_id -- Used in dashboard queries filtering students by branch.
  • students.section_id -- Used in section-scoped dashboard queries.
  • fee_requests.student_id -- Used in JOINs to resolve student details for each request.
  • fee_requests.semester_id -- Used in semester-scoped queries.
  • user_sections.user_id -- Used to resolve which sections a staff member is assigned to.
  • user_sections.section_id -- Used in reverse lookups to find staff assigned to a section.
  • activity_logs.created_at -- Used by the pg_cron cleanup job and time-sorted log views.
Performance Note: At the expected scale of the LTSU Fee Portal (college-level deployment with hundreds of students and a handful of staff), the automatic primary key and unique constraint indexes are sufficient for excellent query performance. The PostgreSQL query planner handles sequential scans on small tables efficiently, and the overhead of maintaining additional indexes may not be justified until the dataset grows beyond several thousand rows per table.
